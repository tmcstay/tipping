## In progress this session (2026-07-13), part 2: GrandTour auto dry-run stage-1 starvation + exit-code + email-gate fix

## In progress this session (2026-07-13), part 4: fixed a real production regression in the auth callback (loop, not just a stuck spinner)

Part 1's fix (above) was deployed to production, but the loop persisted - now visibly *flashing* rather than a static stuck spinner. Full root cause and fix detail is in `CLAUDE.md`'s new "Auth callback routing, part 2" section. Summary:
- **Actual root cause**: two independent things deciding "where to navigate next" at the same moment. A successful `exchangeCodeForSession` updates `AuthProvider`'s session, which flips the root `Stack.Protected` guards (`app/_layout.tsx`) - Expo Router's own docs confirm a guard flip mutates navigation history/can redirect on its own - at essentially the same instant `AuthCallbackScreen` was *also* calling `router.replace`, without waiting for that guard state to settle. If the guard flip forced a remount of `AuthCallbackScreen` while still on that route, its old per-mount `handledRef` reset, and the effect fired again - issuing a **second** `exchangeCodeForSession` call against an already-being-consumed one-time-use code, which fails and re-enters the spinner cycle. `ProtectedRoute` also blanket-gated `/auth/callback` behind `AuthProvider`'s own `loading` flag, adding a second unrelated spinner-flash layer.
- **Fix**: `AuthCallbackScreen.tsx` rewritten around a module-level `flowRegistry` (survives remounts, unlike a `useRef`) keyed by `getAuthCallbackFlowKey` (new, in `authCallbackExperience.ts`) - every mount for the same callback attempt reuses the same in-flight/completed Supabase call, never issues a second one. Two-phase state machine: exchange succeeds → `awaiting_session` (waits for `useAuth()`'s `user` to actually reflect the new session, bounded by a 5s fallback) → redirects exactly once, by construction *after* the same signal that drives the `Stack.Protected` guards has already been observed - no more race. `ProtectedRoute.tsx` now excludes `/auth/callback` (`isAuthCallbackPathname`, new shared helper) from its loading-spinner gate entirely.
- New `apps/mobile/lib/authDebugLog.ts` - `__DEV__`-gated tracing (pathname, param *names* only, loading/session-exists booleans, auth event names, redirect source+destination) added to `AuthCallbackScreen`/`AuthProvider`/`ProtectedRoute`, never active in production.
- **Verified, not changed**: root route never redirects to `/auth/callback` under any auth state; no stale `returnTo`/pathname persistence exists anywhere in the app (`AsyncStorage`'s only use is Supabase's own session storage); logout was already correct from part 1.
- Verified: `apps/mobile` `test:ui` 129/129 passing (was 122; +7 new: `getAuthCallbackFlowKey` dedup/stability, `isAuthCallbackPathname`), root `typecheck` clean, root `npm test` 66+7 passing, `apps/mobile` `web:build` succeeded, `git diff --check` clean.
- Committed as `fc86222`, pushed and deployed - **this fix alone did not resolve the issue.** User reported it made things worse ("any access to the main web page flashes and goes nowhere," not just `/auth/callback`) - see part 5 below for the real fix.

## In progress this session (2026-07-13), part 5: the REAL root cause, found with an actual browser this time

Part 4 was diagnosed from code reading alone (no browser tool was available in this environment). When the user reported it hadn't helped - and had gotten *broader* (every page, not just the callback route) - that breadth was the signal the real cause was structural. This time, Playwright/`chromium` was installed (`npx playwright install chromium --with-deps`, ~300MB, throwaway - never added to this repo's `package.json`) and used to actually load the live production site with full console/error/navigation-event capture. Full detail in `CLAUDE.md`'s new "Auth callback routing, part 3" section. Summary:
- **Bug A (confirmed via browser console):** bare `https://grandtour-three.vercel.app/auth/callback` threw `Error: Attempted to navigate before mounting the Root Layout component.` - a documented Expo Router error from calling `router.replace(...)` imperatively inside a `useEffect` that can fire before the navigator finishes mounting. Part 4's `AuthCallbackScreen` still did this on its very first possible tick (the `redirect_home` case).
- **Bug B (confirmed via `framenavigated` event capture, the actual production loop):** bare `https://grandtour-three.vercel.app/` produced **hundreds of same-URL client-side navigations per second, forever** (`/login` - a single unambiguous route - loaded and settled cleanly in the identical test, isolating the bug to `/` specifically). Root cause: `app/index.tsx` and `app/(auth)/index.tsx` were **both registered for the exact same URL path `/`** (Expo Router route groups don't add a URL segment), differentiated only by which of two complementary `Stack.Protected` guards was active. Two screens permanently competing for one path = a permanent navigation loop.
- **Fix**: `app/(auth)/index.tsx` deleted (now unreachable). `app/_layout.tsx`: `index` moved out of its `Stack.Protected` group, registered unconditionally (same pattern as `auth/callback`) - `/` now has exactly one owner. `app/index.tsx`: default export is now a thin guard (`if (!user || isPasswordRecovery) return <Redirect href="/login" />`) before rendering the real dashboard (renamed `AuthenticatedDashboard`). `AuthCallbackScreen.tsx`: every `router.replace(...)` call replaced with declarative `<Redirect href={destination} />` JSX - Expo Router's own documented-safe pattern, which structurally cannot hit the "before mounting" race.
- **Verified locally with the actual fix, not just typecheck**: rebuilt (`npm run web:build`), served via `serve -s dist` (SPA fallback matching `vercel.json`), drove it with Playwright - `/`, `/auth/callback`, and `/login` all now load and settle in ~7-8 events each (down from 1800+ and climbing), zero `pageerror`s, screenshot confirms a real, correctly-rendered "Welcome back" sign-in screen at `/`.
- Verified: `apps/mobile` `typecheck` clean, `test:ui` 129/129, root `npm test` 66+7, `git diff --check` clean.
- **Files changed this part**: `apps/mobile/app/_layout.tsx`, `apps/mobile/app/index.tsx`, `apps/mobile/screens/AuthCallbackScreen.tsx`, `apps/mobile/app/(auth)/index.tsx` (deleted), `CLAUDE.md`, `HANDOFF.md`.
- **Lesson for future auth/routing work in this app**: this environment has no browser tool by default, and two rounds of pure code-reading diagnosis both missed the real cause. Install Playwright's chromium (throwaway, scratchpad-only) and actually load the page before trusting a routing/redirect fix.
