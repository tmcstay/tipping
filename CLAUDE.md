## Project

`tipping-suite` — a tipping/prediction app. This file currently documents only the **GrandTour (Tour de France) official-letour feed pipeline**, which is the area actively under development on branch `feature/grandtour-official-letour-provider`. Other parts of the app (auth, other sports/competitions) are not yet documented here.

## GrandTour official-letour feed pipeline

### Purpose
Import official Tour de France stage results and jersey holders from letour.fr into Supabase, safely, with a fully-gated dry-run → reconciliation → apply → admin-review → finalize → score flow. Everything defaults to read-only; writes require explicit multi-flag confirmation, and every write step (apply, admin-check, finalize) is a separate, independently-gated action. Apply, admin-check, and finalize are all callable by service_role (CLI) **or** by an authenticated cycling admin's own session (admin UI, since `20260714010000_grandtour_apply_authenticated_grant.sql` extended apply the same way `20260710060000` already extended admin-check/finalize) — score has always required a real authenticated admin session, never service_role. No route or RPC in this pipeline ever accepts/uses a service-role key from the browser or from Vercel.

### Pipeline stages
1. **Parse** (`scripts/grandtour-feed-provider.mjs`) — fetches `https://www.letour.fr/en/rankings/stage-<N>` and regex-parses:
   - The stage-result ranking table (`rankingTable` / `rankingTables__row` markup) → top finishers with position, name, bib, team, time, gap. This markup is confirmed against a real letour.fr page.
   - Jersey holders: **not** on the initial page. The four cumulative classifications that determine jersey holders (yellow=individual/GC, green=points, kom=climber, white=youth) live behind a "General ranking" tab whose four sub-tab URLs are embedded as a `data-ajax-stack` JSON attribute on the page (`extractGeneralClassificationAjaxUrls`). `fetchLetourJerseyHolders` scrapes those URLs fresh per stage, fetches each fragment (same `rankingTable` row markup, confirmed real), and takes the position-1 leader of each (`parseLetourClassificationLeader`). This is confirmed against real letour.fr markup (verified mid-session; the URL hashes are per-page-load tokens and must always be scraped fresh, never hardcoded).
   - `OfficialLetourGrandTourFeedProvider` orchestrates one main-page fetch + up to 4 jersey-classification fetches per stage, and attaches `jersey_holders` + `jersey_fetch_metadata` diagnostics (status: `found`/`table_not_found`/`empty_table`/`fetch_error`/`unsupported_markup`) to each parsed `stageResult`.
2. **Reconcile** (`scripts/grandtour-reconciliation.mjs` + `grandtour-reconciliation-supabase.mjs`) — read-only (anon key), matches parsed riders/teams against `grandtour_riders`/`grandtour_teams`/`grandtour_stage_startlists` by bib number first, then normalized name. Produces `reconcileStageResult()` output including `matchedRiders`, `jerseyHolders`, `safeToApply`, and `blockers`.
3. **Apply** — two paths, same RPC. CLI: `scripts/grandtour-apply.mjs` + `scripts/grandtour-feed-import.mjs`'s `runApply` reads a previously-generated `--reconcile` report from disk (never re-fetches live), validates every gate, then calls `apply_grandtour_official_stage_result` via `SUPABASE_SERVICE_ROLE_KEY`. Admin UI: the stage card's "Apply Official Result" button (only enabled after a passing "Run Official Check") calls `apps/mobile/api/admin/grandtour/apply-official-result.mjs`, which fetches a **fresh** report server-side (never trusts a client-supplied one), re-validates it with the exact same `scripts/grandtour-apply.mjs` functions the CLI uses, then calls the same RPC using the admin's own session (never a service-role key). Both paths write a **draft** (`is_final: false`, `review_status: 'imported'`, `source_mode: 'official_feed'`) `grandtour_stage_results` + `grandtour_stage_result_lines` (top-10 only) + `grandtour_stage_jersey_holders` (all 4, if provided). Never scores, never finalizes.
4. **Admin check** (`public.mark_grandtour_stage_result_checked` RPC — CLI: `scripts/grandtour-admin-stage.mjs --mark-checked`; UI: `/admin/grandtour-stages` "Mark Checked"; or direct SQL/service-role) — a human confirms the applied draft is correct. Sets `review_status: 'admin_checked'`. Required before finalize will accept the stage.
5. **Finalize** (`public.finalize_grandtour_stage_result` RPC) — sets `is_final: true`, `review_status: 'finalised'`. Refuses unless `review_status = 'admin_checked'` first. Never scores, never touches result lines/jersey holders.
6. **Score** (`public.recalculate_grandtour_stage_scores` — pre-existing, unmodified) — requires an **authenticated cycling-admin session** (not service_role — it's `security invoker` and checks `auth.uid()` against `user_app_memberships`). Only works once `is_final = true`.

### Database (GrandTour-relevant tables)
- `grandtour_teams`, `grandtour_riders` (with `bib_number`, `is_active`, `status`/`status_changed_at`/`status_reason`, `specialities`), `grandtour_stage_startlists` (`status`: provisional/confirmed/withdrawn/reserve/dns/dnf/otl/suspended/excluded/unknown).
- `grandtour_stage_results` — `is_final` boolean, plus (new): `review_status` enum (`draft`/`imported`/`review_required`/`admin_checked`/`finalised`/`correction_required`), `admin_checked_at`/`admin_checked_by`/`admin_check_note`, `finalised_at`/`finalised_by`/`finalisation_reason`, `source_mode` enum (`official_feed`/`manual_admin`/`mixed`/`test`). Check constraint: `is_final = (review_status = 'finalised')` — both must move together.
- `grandtour_stage_result_lines` (top-10, unique per stage+position and stage+rider).
- `grandtour_stage_jersey_holders` (`stage_id`, `jersey_type` enum: yellow/green/kom/white, `rider_id`; unique on `(stage_id, jersey_type)`). Triggers `validate_jersey_holder()` (must be on stage startlist; stage result must exist and not be final) and `prevent_final_jersey_holder_delete()`.
- `grand_tours` — added `manual_result_entry_enabled boolean default false` (admin-controlled gate; no manual-entry RPC exists yet to read it — see Known gaps).
- `grandtour_feed_import_runs` / `grandtour_feed_snapshots` — audit trail written by the apply RPC (pre-existing).
- **New**: `public.grandtour_result_audit_log` (id, stage_id, stage_result_id, action, changed_by, reason, before_payload, after_payload, created_at) — append-only (blocked by trigger), RLS-read-gated to cycling admins. `action` check constraint: `official_import_applied`, `manual_result_created`, `manual_result_updated`, `jersey_holder_updated`, `admin_checked`, `finalised`, `unfinalised`, `scored` (only the first three action values are currently written by real code: `official_import_applied` by apply, `admin_checked`/`finalised` by their RPCs — the rest are reserved for the not-yet-built manual-entry/unfinalize/scoring-audit paths). Distinct from the pre-existing generic `grandtour_game_audit` trigger-based trail (still fires independently on every raw row mutation for a different purpose — both exist, on purpose).
- **Trigger gotcha (fixed, keep in mind for future append-only tables):** `grandtour_result_audit_log`'s FK columns (`stage_result_id`, `changed_by`) use `on delete set null`. An append-only trigger that unconditionally blocks all `UPDATE`s will also block that legitimate FK-driven `SET NULL` action when the referenced row is deleted — breaking any documented delete/rollback path for the parent table. The fix: explicitly allow updates that *only* null out an FK column and change nothing else; reject everything else.

### RPCs
- `public.apply_grandtour_official_stage_result(p_stage_id, p_result_lines, p_reconciliation, p_dry_run_status, p_source, p_finalize, p_reason, p_request_id, p_jersey_holders)` returns jsonb — `security definer`. Re-validates the reconciliation payload server-side, refuses TTT stages, refuses `p_finalize: true` unconditionally, upserts jersey holders via `on conflict (stage_id, jersey_type) do update`, idempotent on unchanged result-line content (`no_change` vs `applied`). Tags new rows `review_status='imported'`, `source_mode='official_feed'`; logs one `official_import_applied` row to `grandtour_result_audit_log` on a genuine `applied` outcome (not on `no_change`; `changed_by` on that row is always `null` regardless of caller — there's no `p_applied_by` parameter, unlike mark-checked/finalize's explicit `p_checked_by`/`p_finalized_by` — a known, intentionally-left-alone gap). **Callable by `service_role` (CLI) or by an authenticated cycling admin's own session (admin UI) since `20260714010000_grandtour_apply_authenticated_grant.sql`** (same `auth.role() = 'service_role' or is_cycling_admin()` guard pattern as mark-checked/finalize, added to the existing byte-identical 9-arg signature — a safe same-OID `create or replace`, no drop needed).
- `public.mark_grandtour_stage_result_checked(p_stage_id, p_checked_by, p_note, p_request_id)` returns jsonb — `security definer`. Requires exactly 10 result lines + 4 jersey holders, non-TTT, a draft result must exist. Sets `review_status='admin_checked'`. Never scores, never finalizes. `p_checked_by` is FK'd to `auth.users(id)`. **Callable by `service_role` (no internal check needed) or by `authenticated` if `grandtour_private.is_cycling_admin()` is true** (internal guard: `auth.role() = 'service_role' or is_cycling_admin()`, added by `20260710060000_grandtour_admin_check_finalise_authenticated_grants.sql` alongside the `authenticated` EXECUTE grant, so the admin UI can call it directly without a service-role key).
- `public.finalize_grandtour_stage_result(p_stage_id, p_finalized_by, p_reason, p_request_id)` returns jsonb — `security definer`. **Signature changed mid-project** (originally 3-arg `(p_stage_id, p_reason, p_request_id)` with no review-status gate; redesigned to require `review_status='admin_checked'` first and take an explicit `p_finalized_by`). Sets `is_final=true`, `review_status='finalised'`. Idempotent (`no_change` if already final). Never scores. Same `service_role`-or-admin-`authenticated` grant/guard as `mark_grandtour_stage_result_checked` above, added by the same migration.
- `public.set_grandtour_manual_result_entry_enabled(p_grand_tour_id, p_enabled, p_changed_by, p_reason)` returns jsonb — `security definer`, `service_role`-only. Flips `grand_tours.manual_result_entry_enabled`; logs to `grandtour_game_audit` (`action='admin_override'`) since it's tour-scoped, not stage-scoped.
- `public.recalculate_grandtour_stage_scores(...)` — pre-existing, unmodified. `security invoker`, requires an authenticated cycling-admin session (checks `grandtour_private.is_cycling_admin()` via `auth.uid()`) — **the service-role key alone cannot call this**, unlike every other RPC above.
- `public.is_current_user_cycling_admin()` returns boolean — `security definer`, `set search_path = ''`. Added by `20260713010000_grandtour_is_current_user_cycling_admin_rpc.sql`. A public wrapper around the already-fixed `grandtour_private.is_cycling_admin()` (that private schema isn't exposed via PostgREST, so it couldn't be called directly). Granted to `authenticated` only — Postgres grants EXECUTE to `PUBLIC` by default on a new function, so this migration explicitly `revoke all ... from public` first, then grants to `authenticated` (the same gotcha bit the local SQL test on first pass: `anon` could otherwise call it despite no explicit grant). Read-only, no writes. Used server-side by `apps/mobile/api/admin/grandtour/run-official-check.mjs` to authorize before running any check, since that route has no service-role key available to it.
- `public.correct_grandtour_stage_result_from_reviewed_report(p_stage_id, p_result_lines, p_jersey_holders, p_reconciliation, p_reason, p_request_id)` returns jsonb — `security definer`. Added by `20260711010000_grandtour_correct_stage_result_rpc.sql` (Part C). Same `service_role`-or-admin-`authenticated` guard as mark-checked/finalise. **The only RPC allowed to touch an existing (draft or already-finalised/scored) result** — `apply_grandtour_official_stage_result` still refuses outright if a different draft or any final result already exists; this is the deliberate, audited bypass of that refusal. Refuses a blank `p_reason`, a stage with no existing result at all (use apply instead), TTT stages, and every apply-style reconciliation/line/jersey-holder gate. On a real content change: unfinalises first (required — see the trigger-ordering gotcha below), always lands on `review_status='correction_required'`/`is_final=false`, deletes any existing `grandtour_stage_scores` rows for the stage and moves affected tips from `'scored'` to `'corrected'` (never deletes/reverts them further — `recalculate_grandtour_stage_scores` already keeps a `'corrected'` tip's status as `'corrected'`, not `'scored'`, after rescoring), and logs one `result_corrected` row to `grandtour_result_audit_log`. Idempotent no-op (`status='no_change'`, no writes at all) on byte-identical content. Never finalises, never scores. CLI: `scripts/grandtour-admin-stage.mjs --update-results --from-report <path> --reason <text>` (never fetches letour.fr itself — same `--from-report`-only trust boundary as `--apply`). UI: `/admin/grandtour-stages`' "Update Results / Re-run Official Import" panel (paste a CLI-generated report; the UI cannot fetch letour.fr itself — no server-side scraping endpoint exists in this project). Full workflow: `docs/GRANDTOUR_APPLY_MODE_PRODUCTION_READINESS_CHECKLIST.md` §15.

### Postgres gotchas learned this session (apply broadly)
- Adding a parameter to a `security definer` function via a plain `create or replace function` is **not** a same-OID replace if the parameter list changes — Postgres treats a different arg-type list as a distinct overload, silently leaving both old and new versions defined. Whenever a function's parameter list changes (not just its body), explicitly `drop function if exists ...(<old exact signature>)` before recreating it. `create or replace function` is only safe as a plain in-place update when the parameter list is byte-identical.
- **A `returns boolean` SQL function whose body is `select <expr> from ... where exists (...)` returns SQL `NULL` for the "false" case, not `false`** — if the `where exists(...)` filters out the row, the query returns *zero rows*, and a zero-row scalar-context function call evaluates to NULL. This silently broke `grandtour_private.is_cycling_admin()`: every `if not is_cycling_admin() then raise exception ...` guard (imperative PL/pgSQL) was a no-op for non-admins, because `if not NULL` is `NULL`, and `if` only takes the branch when the condition is `TRUE`. Confirmed via direct testing that a non-admin authenticated session could call `recalculate_grandtour_stage_scores()` and sail straight past the intended admin check. **RLS `using(...)`/`with check(...)` policies were unaffected** — Postgres RLS treats NULL the same as false — only the `if not X then raise` pattern was exposed. Fixed in `20260710060000_grandtour_admin_check_finalise_authenticated_grants.sql` by rewriting the function body as `select exists (...)`, which always returns a genuine `true`/`false`, never NULL. **Lesson: any `returns boolean` SQL function meant to be used in an imperative `if not fn() then raise` guard must be written as `select exists(...)` (or with an explicit `coalesce(..., false)`), never as a bare `where exists(...)`-filtered row select** — write a genuine negative-path test (a real non-admin session actually calling the guarded function) before trusting *any* such guard, not just a positive-path (admin-succeeds) test.
- **`grandtour_stage_result_lines`/`grandtour_stage_jersey_holders` both have "prevent final delete" triggers** (`grandtour_stage_result_lines_prevent_final_delete`/`grandtour_stage_jersey_holders_prevent_final_delete`, from `20260629080958_grandtour_mvp.sql`) that block `DELETE` while the parent `grandtour_stage_results.is_final` is true. `correct_grandtour_stage_result_from_reviewed_report` (Part C) therefore flips `is_final` to `false` **first**, in the same function call, before deleting/reinserting any line or jersey-holder row — reversing that order makes correcting an already-finalised stage fail outright with an opaque trigger error. Same applies to `grandtour_stage_team_result_lines` (TTT) via `prevent_final_team_result_line_delete`, not currently exercised since TTT corrections are out of scope.

### Safety conventions (load-bearing — don't relax without explicit instruction)
- **Dry-run by default everywhere.** `--apply` requires `--confirm-provider official-letour`, `--confirm-stage <N>`, and (against a known production URL) `--confirm-production`.
- Apply mode **never** re-fetches or re-reconciles live — it only trusts a `--from-report` file, checked for freshness (max 6 hours old) and full gate compliance (`validateReportForApply`).
- Apply is **one-stage-only**, **top-10-only**, and writes **drafts only**. Admin-check, finalize, and score are separate, later, independently-approved steps — never bundle them into the same operator sitting as apply without a fresh review.
- `SUPABASE_SERVICE_ROLE_KEY`'s JWT `role` claim is decoded and checked (`decodeJwtRole`) before any apply — refuses to run with an anon key even if misconfigured.
- Known production project ref: `nsdpilmmrfobiapbwona` (`KNOWN_PRODUCTION_PROJECT_REFS` in `scripts/grandtour-apply.mjs`).
- The scheduled GitHub Actions workflow (`.github/workflows/grandtour-daily-feed-dry-run.yml`) is dry-run only — no `SUPABASE_SERVICE_ROLE_KEY`, no `--apply`, ever.

### Team-name normalization
`normalizeTeamName` (in `scripts/tdf-data-utils.mjs`) has its own implementation (distinct from `normalizeRiderName`) that folds punctuation/hyphens/pipes to whitespace before comparing, so official-letour's compact uppercase names (`"LIDL-TREK"`) correctly equate to canonical DB names (`"Lidl - Trek"`). `normalizeRiderName` deliberately does not do this (hyphenated surnames must not be split).

### Known gaps / follow-ups
- **Manual result entry is designed but not implemented.** `grand_tours.manual_result_entry_enabled` and `set_grandtour_manual_result_entry_enabled` exist and work, but nothing reads the flag — there's no RPC yet for an admin to hand-enter/correct a stage result when the feed fails. Full intended contract is documented in `docs/GRANDTOUR_APPLY_MODE_PRODUCTION_READINESS_CHECKLIST.md` §14.5.
- **No "unfinalise" RPC.** `'unfinalised'` is reserved in `grandtour_result_audit_log`'s action list but nothing writes it. A final result cannot currently be reopened via RPC (only by mistake-prone direct table access, which auto-marks a game-audit trail via the pre-existing generic trigger but not the new stage-workflow one).
- **`recalculate_grandtour_stage_scores` does not write to the new `grandtour_result_audit_log`.** It keeps using its own pre-existing `grandtour_game_audit` trail (`action='score_calculated'`/`'score_recalculated'`). `'scored'` is reserved in the new table's action list but currently unwritten — left alone deliberately (it's a well-tested pre-existing RPC, out of this session's explicit scope).
- No production-readiness checklist exists for `scripts/load-tdf-2026-startlist.mjs` specifically (unlike the apply-RPC one). It also has no `decodeJwtRole`-style key-type guard.
- No correction workflow exists for an already-applied draft result with *different* content — the apply RPC hard-refuses and requires manual admin-RLS-path correction or delete+reapply (the documented rollback: `delete from grandtour_stage_results where id = ...`, which now correctly cascades/nulls the audit log rather than being blocked — see the trigger gotcha above).

## Working conventions in this repo
- Local Supabase: `npx supabase status` for connection details; local demo anon/service-role keys are well-known Supabase CLI defaults, not secrets.
- Always rehearse against **local Supabase only** before any production action; production actions require explicit user authorization in the task itself, and even then double-check any pasted key's `role` claim (JWT) before using it — don't assume a pasted "anon key" or "production key" is actually what it's labeled.
- `npm run test:data` runs all `scripts/*.test.mjs` via Node's built-in test runner. `supabase/tests/*.sql` are pgTAP-style scripts run via `docker exec ... psql -f`, each in its own `begin`/`rollback` transaction. `npm run grandtour:reconcile:local-smoke`, `npm run grandtour:apply:local-smoke`, and `npm run grandtour:admin-stage:local-smoke` are real-Supabase (local) end-to-end smoke tests, not mocked — the apply and admin-stage ones perform real writes and clean up after themselves (including any test `auth.users` rows they create via `signUp`). The admin-stage smoke test's `--grand-tour-name`/`--grand-tour-year` fixture values must match `supabase/seed.sql`'s local grand tour (`"GrandTour France 2026"`, 2026) — the CLI's own real-world default (`"Tour de France"`) is a production concept, not a local one. From `apps/mobile`: `npm run test:ui` (pure UI logic, compiled via `tsc` into `dist/mobile-tests` then run with `node --test`) and `npm run test:api` (the `apps/mobile/api/**/*.mjs` server routes, `node --test` directly — no `tsc` step, since `apps/mobile/tsconfig.json`'s `include` deliberately excludes `api/`).
- **Node-in-CI gotcha:** `@supabase/supabase-js`'s `createClient()` throws synchronously on Node 20 (`Node.js 20 detected without native WebSocket support`) because its Realtime client requires a WebSocket constructor and no `ws` package is installed as a fallback in this repo. Any GitHub Actions workflow that constructs a Supabase client (i.e. runs `--reconcile`, or anything else that calls `createClient`) needs `actions/setup-node@v4` with `node-version: '22'` or higher — `grandtour-auto-dry-run.yml` hit this on its first real run (fixed by bumping from `'20'` to `'22'`). `grandtour-daily-feed-dry-run.yml` never calls `--reconcile` so it was never exposed to this, but keep it in mind for any future workflow that does.
- **Postgres gotcha:** a newly-created function gets EXECUTE granted to `PUBLIC` by default. `grant execute on function ... to authenticated` alone does **not** revoke that — `anon` (or anyone) can still call it. Any function meant to be admin/authenticated-only needs an explicit `revoke all on function ... from public;` before the intended grant (see `20260713010000_grandtour_is_current_user_cycling_admin_rpc.sql`, and the SQL test that actually caught this by exercising the `anon` role, not just `authenticated`).
- **PL/pgSQL gotcha for test authors:** an `exception when others` block implicitly rolls back to a savepoint taken at the *start* of that block — so a deliberate `DELETE`/setup statement meant to persist past the exception must be a separate top-level statement, never inside the same `do $$ ... exception ... $$` block being tested.
- Docs for this pipeline live in `docs/grandtour-results-feed.md` (central operational doc), `docs/grandtour-apply-mode-spec.md` (design spec), `docs/GRANDTOUR_PRODUCTION_DEPLOYMENT_CHECKLIST.md` (schema/migration deploys), `docs/GRANDTOUR_APPLY_MODE_PRODUCTION_READINESS_CHECKLIST.md` (full apply → admin-check → finalize → score production runbook, §14 covers the review workflow), `docs/GRANDTOUR_TDF_2026_STARTLIST_PRODUCTION_LOAD.md` (production load record).
