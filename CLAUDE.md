## Dashboard, stage closure state, and result eligibility

### Stage closure state (open/closing_soon/closed/live/completed)
- `packages/tipping-core/src/cycling-stage-tip.ts` exports `resolveCyclingStageClosureState(input)` тАФ the single source of truth for whether a stage is currently open for tipping. Previously the mobile app hand-rolled `now >= stage.locks_at` three different ways (dashboard, `LockCountdownCard`, stages list), each with slightly different null-handling and **none of them read `grandtour_stages.manual_locked_at`** (an admin override) at all. This function fixes both: it extends `CyclingStageLockInput`/`resolveCyclingStageLockAt`/`isCyclingStageTipLocked` with `locksAt`/`manualLockedAt` fields (priority `manualLockedAt > locksAt > startTime(legacy) > stageDate+defaultLockTimeUtc`, additive/backward-compatible), then layers `live` (now >= `starts_at`) and `completed` (`isFinal === true`) around that boolean, in that priority order (`completed` > `live` > the lock-derived `closed`/`closing_soon`/`open`). `closing_soon` is <24h before lock. A single `now` is resolved once per call and reused throughout тАФ callers must do the same (resolve `now = new Date()` once per render, never call `new Date()` repeatedly while building one card).
- `apps/mobile/lib/stageClosureExperience.ts`'s `buildClosureDisplay(state, ...)` turns that resolved state into exactly what a card renders: badge label, primary text line, emphasis (true for live, and for closing_soon under 60 minutes), whether tip-editing is allowed, and the CTA label. Deliberately takes the state as a plain string union (not a runtime import of the tipping-core type) and a pre-formatted date/time string тАФ this file is compiled standalone by `apps/mobile/package.json`'s `test:ui` script (see below), which cannot resolve cross-package runtime imports.
- `apps/mobile/components/StageStatusBadge.tsx` тАФ the compact Open/Closing soon/Closed/Live/Provisional/Completed pill used across the dashboard.
- `apps/mobile/components/LockCountdownCard.tsx` and `apps/mobile/app/stages/index.tsx` both now call `resolveCyclingStageClosureState` instead of their own inline date math. `apps/mobile/app/stages/[stageId].tsx` (the actual tip-entry screen) was **deliberately left untouched** тАФ it has its own, more complex, mode-aware lock logic (`daily` vs `preselection`, reading `race.preselection_locks_at`) that doesn't map onto the single-stage closure model above; touching the highest-stakes write screen was out of scope for this pass.

### Stage-result eligibility ("future stages must never appear as results")
- `packages/tipping-core/src/stage-eligibility.ts` exports `isStageEligibleForResults(stage, now)` and `selectLatestEligibleStage(candidates, now)`. The real schema has no `stage.status`/`result.result_status` columns matching a generic in_progress/completed/provisional/official vocabulary тАФ this adapts the conceptual rule to what actually exists: `stage.starts_at <= now AND (result.is_final === true OR result.review_status is one of imported/review_required/admin_checked/correction_required)`. A stage row existing (or even a `draft` result existing) is never treated as a real result on its own.
- **Real constraint, not a gap to "fix" later:** the public app's anon/publishable key can only ever see `grandtour_stage_results` rows where `is_final = true` (RLS тАФ see the GrandTour pipeline section above). So `reviewStatus`/"provisional" support in the predicate exists for correctness and any future/admin-authenticated caller, but the public dashboard today will only ever pass `isFinal`, never a genuine `provisional` result тАФ this is expected, not a bug.
- Applied at three layers, per the task's "earliest reliable layer" requirement: (1) `packages/supabase-client/src/cycling.ts`'s `listCyclingStageResults`/`getCyclingStageResult` now also filter `.lte("grandtour_stages.starts_at", now)` at the query layer (both took a new optional `now: Date = new Date()` param, backward compatible); (2) the shared `isStageEligibleForResults`/`selectLatestEligibleStage` selectors, called directly from `apps/mobile/app/index.tsx` and `apps/mobile/app/results.tsx`; (3) a final defensive filter in those same screens. "Latest completed stage" is always computed via `selectLatestEligibleStage` (sorted by actual `starts_at` descending, stage number as a deterministic tie-breaker) тАФ never an unordered client-side `.find`/re-sort-by-stage_number, which is what both screens did independently before (and which had no time-based guard at all).

### Dashboard (`apps/mobile/app/index.tsx`)
Restructured into a fixed hierarchy: (1/2) a single hero card that shows either "action required" (soonest stage still open/closing_soon for tipping) or, if nothing is open, a currently-live stage тАФ merged into one card rather than two, since in this one-stage-per-day model they're normally the same stage in different states; (3) latest eligible result; (4) user competition summary (rank/behind-leader/last-stage stats, jersey holders, mini leaderboard); (5) upcoming stages (excluding whatever's already shown as the hero); (6) recent results (excluding the stage already shown in position 3). Added loading skeletons (`SkeletonCard`/`SkeletonStatGrid` in `apps/mobile/components/DataState.tsx`, sized to match the real cards so nothing jumps on load) and error/empty states тАФ the dashboard previously had none at all.

### Reusable clickable card
`apps/mobile/components/InfoCard.tsx` (the shared card chrome nearly every screen already used) gained optional `href`/`onPress`/`accessibilityLabel`/`accessibilityHint`/`chevron` props rather than a new parallel component. `href` renders via `expo-router`'s `Link` with `asChild` (real `<a href>` semantics, keyboard focus, and cmd/ctrl-click-to-open-in-new-tab on web), wrapping a `Pressable` with `accessibilityRole="button"` and a pressed-state opacity change; `chevron` defaults to on whenever the card is interactive. `DashboardStatCard.tsx` and `JerseyHolderCard.tsx` got the same optional `href` treatment directly (kept as their own lightweight components rather than rebuilt on `InfoCard`, since their internal layout genuinely differs). All three are fully backward compatible тАФ every other existing call site across the app (which never passes `href`/`onPress`) renders identically to before. `apps/mobile/lib/stageClosureExperience.ts` also exports small pure `build*DashboardCardLink` functions (href + accessibility label/hint per card type) so navigation targets and labels are unit-tested the same way the rest of this app's view-model logic is (no React rendering test harness exists in this repo, and none was added тАФ see Testing conventions below).

## GrandTour official-letour feed pipeline

### Purpose
Import official Tour de France stage results and jersey holders from letour.fr into Supabase, safely, with a fully-gated dry-run тЖТ reconciliation тЖТ apply тЖТ admin-review тЖТ finalize тЖТ score flow. Everything defaults to read-only; writes require explicit multi-flag confirmation, and every write step (apply, admin-check, finalize) is a separate, independently-gated action. Apply, admin-check, and finalize are all callable by service_role (CLI) **or** by an authenticated cycling admin's own session (admin UI, since `20260714010000_grandtour_apply_authenticated_grant.sql` extended apply the same way `20260710060000` already extended admin-check/finalize) тАФ score has always required a real authenticated admin session, never service_role. No route or RPC in this pipeline ever accepts/uses a service-role key from the browser or from Vercel.

### Pipeline stages
1. **Parse** (`scripts/grandtour-feed-provider.mjs`) тАФ fetches `https://www.letour.fr/en/rankings/stage-<N>` and regex-parses:
   - The stage-result ranking table (`rankingTable` / `rankingTables__row` markup) тЖТ top finishers with position, name, bib, team, time, gap. This markup is confirmed against a real letour.fr page.
   - Jersey holders: **not** on the initial page. The four cumulative classifications that determine jersey holders (yellow=individual/GC, green=points, kom=climber, white=youth) live behind a "General ranking" tab whose four sub-tab URLs are embedded as a `data-ajax-stack` JSON attribute on the page (`extractGeneralClassificationAjaxUrls`). `fetchLetourJerseyHolders` scrapes those URLs fresh per stage, fetches each fragment (same `rankingTable` row markup, confirmed real), and takes the position-1 leader of each (`parseLetourClassificationLeader`). This is confirmed against real letour.fr markup (verified mid-session; the URL hashes are per-page-load tokens and must always be scraped fresh, never hardcoded).
   - `OfficialLetourGrandTourFeedProvider` orchestrates one main-page fetch + up to 4 jersey-classification fetches per stage, and attaches `jersey_holders` + `jersey_fetch_metadata` diagnostics (status: `found`/`table_not_found`/`empty_table`/`fetch_error`/`unsupported_markup`) to each parsed `stageResult`.
2. **Reconcile** (`scripts/grandtour-reconciliation.mjs` + `grandtour-reconciliation-supabase.mjs`) тАФ read-only (anon key), matches parsed riders/teams against `grandtour_riders`/`grandtour_teams`/`grandtour_stage_startlists` by bib number first, then normalized name. Produces `reconcileStageResult()` output including `matchedRiders`, `jerseyHolders`, `safeToApply`, and `blockers`.
3. **Apply** тАФ two paths, same RPC. CLI: `scripts/grandtour-apply.mjs` + `scripts/grandtour-feed-import.mjs`'s `runApply` reads a previously-generated `--reconcile` report from disk (never re-fetches live), validates every gate, then calls `apply_grandtour_official_stage_result` via `SUPABASE_SERVICE_ROLE_KEY`. Admin UI: the stage card's "Apply Official Result" button (only enabled after a passing "Run Official Check") calls `apps/mobile/api/admin/grandtour/apply-official-result.mjs`, which fetches a **fresh** report server-side (never trusts a client-supplied one), re-validates it with the exact same `scripts/grandtour-apply.mjs` functions the CLI uses, then calls the same RPC using the admin's own session (never a service-role key). Both paths write a **draft** (`is_final: false`, `review_status: 'imported'`, `source_mode: 'official_feed'`) `grandtour_stage_results` + `grandtour_stage_result_lines` (top-10 only) + `grandtour_stage_jersey_holders` (all 4, if provided). Never scores, never finalizes.
4. **Admin check** (`public.mark_grandtour_stage_result_checked` RPC тАФ CLI: `scripts/grandtour-admin-stage.mjs --mark-checked`; UI: `/admin/grandtour-stages` "Mark Checked"; or direct SQL/service-role) тАФ a human confirms the applied draft is correct. Sets `review_status: 'admin_checked'`. Required before finalize will accept the stage.
5. **Finalize** (`public.finalize_grandtour_stage_result` RPC) тАФ sets `is_final: true`, `review_status: 'finalised'`. Refuses unless `review_status = 'admin_checked'` first. Never scores, never touches result lines/jersey holders.
6. **Score** (`public.recalculate_grandtour_stage_scores` тАФ pre-existing, unmodified) тАФ requires an **authenticated cycling-admin session** (not service_role тАФ it's `security invoker` and checks `auth.uid()` against `user_app_memberships`). Only works once `is_final = true`.

### Database (GrandTour-relevant tables)
- `grandtour_teams`, `grandtour_riders` (with `bib_number`, `is_active`, `status`/`status_changed_at`/`status_reason`, `specialities`), `grandtour_stage_startlists` (`status`: provisional/confirmed/withdrawn/reserve/dns/dnf/otl/suspended/excluded/unknown).
- `grandtour_stage_results` тАФ `is_final` boolean, plus (new): `review_status` enum (`draft`/`imported`/`review_required`/`admin_checked`/`finalised`/`correction_required`), `admin_checked_at`/`admin_checked_by`/`admin_check_note`, `finalised_at`/`finalised_by`/`finalisation_reason`, `source_mode` enum (`official_feed`/`manual_admin`/`mixed`/`test`). Check constraint: `is_final = (review_status = 'finalised')` тАФ both must move together.
- `grandtour_stage_result_lines` (top-10, unique per stage+position and stage+rider).
- `grandtour_stage_jersey_holders` (`stage_id`, `jersey_type` enum: yellow/green/kom/white, `rider_id`; unique on `(stage_id, jersey_type)`). Triggers `validate_jersey_holder()` (must be on stage startlist; stage result must exist and not be final) and `prevent_final_jersey_holder_delete()`.
- `grand_tours` тАФ added `manual_result_entry_enabled boolean default false` (admin-controlled gate; no manual-entry RPC exists yet to read it тАФ see Known gaps).
- `grandtour_feed_import_runs` / `grandtour_feed_snapshots` тАФ audit trail written by the apply RPC (pre-existing).
- **New**: `public.grandtour_result_audit_log` (id, stage_id, stage_result_id, action, changed_by, reason, before_payload, after_payload, created_at) тАФ append-only (blocked by trigger), RLS-read-gated to cycling admins. `action` check constraint: `official_import_applied`, `manual_result_created`, `manual_result_updated`, `jersey_holder_updated`, `admin_checked`, `finalised`, `unfinalised`, `scored` (only the first three action values are currently written by real code: `official_import_applied` by apply, `admin_checked`/`finalised` by their RPCs тАФ the rest are reserved for the not-yet-built manual-entry/unfinalize/scoring-audit paths). Distinct from the pre-existing generic `grandtour_game_audit` trigger-based trail (still fires independently on every raw row mutation for a different purpose тАФ both exist, on purpose).
- **Trigger gotcha (fixed, keep in mind for future append-only tables):** `grandtour_result_audit_log`'s FK columns (`stage_result_id`, `changed_by`) use `on delete set null`. An append-only trigger that unconditionally blocks all `UPDATE`s will also block that legitimate FK-driven `SET NULL` action when the referenced row is deleted тАФ breaking any documented delete/rollback path for the parent table. The fix: explicitly allow updates that *only* null out an FK column and change nothing else; reject everything else.

### RPCs
- `public.apply_grandtour_official_stage_result(p_stage_id, p_result_lines, p_reconciliation, p_dry_run_status, p_source, p_finalize, p_reason, p_request_id, p_jersey_holders)` returns jsonb тАФ `security definer`. Re-validates the reconciliation payload server-side, refuses TTT stages, refuses `p_finalize: true` unconditionally, upserts jersey holders via `on conflict (stage_id, jersey_type) do update`, idempotent on unchanged result-line content (`no_change` vs `applied`). Tags new rows `review_status='imported'`, `source_mode='official_feed'`; logs one `official_import_applied` row to `grandtour_result_audit_log` on a genuine `applied` outcome (not on `no_change`; `changed_by` on that row is always `null` regardless of caller тАФ there's no `p_applied_by` parameter, unlike mark-checked/finalize's explicit `p_checked_by`/`p_finalized_by` тАФ a known, intentionally-left-alone gap). **Callable by `service_role` (CLI) or by an authenticated cycling admin's own session (admin UI) since `20260714010000_grandtour_apply_authenticated_grant.sql`** (same `auth.role() = 'service_role' or is_cycling_admin()` guard pattern as mark-checked/finalize, added to the existing byte-identical 9-arg signature тАФ a safe same-OID `create or replace`, no drop needed).
- `public.mark_grandtour_stage_result_checked(p_stage_id, p_checked_by, p_note, p_request_id)` returns jsonb тАФ `security definer`. Requires exactly 10 result lines + 4 jersey holders, non-TTT, a draft result must exist. Sets `review_status='admin_checked'`. Never scores, never finalizes. `p_checked_by` is FK'd to `auth.users(id)`. **Callable by `service_role` (no internal check needed) or by `authenticated` if `grandtour_private.is_cycling_admin()` is true** (internal guard: `auth.role() = 'service_role' or is_cycling_admin()`, added by `20260710060000_grandtour_admin_check_finalise_authenticated_grants.sql` alongside the `authenticated` EXECUTE grant, so the admin UI can call it directly without a service-role key).
- `public.finalize_grandtour_stage_result(p_stage_id, p_finalized_by, p_reason, p_request_id)` returns jsonb тАФ `security definer`. **Signature changed mid-project** (originally 3-arg `(p_stage_id, p_reason, p_request_id)` with no review-status gate; redesigned to require `review_status='admin_checked'` first and take an explicit `p_finalized_by`). Sets `is_final=true`, `review_status='finalised'`. Idempotent (`no_change` if already final). Never scores. Same `service_role`-or-admin-`authenticated` grant/guard as `mark_grandtour_stage_result_checked` above, added by the same migration.
- `public.set_grandtour_manual_result_entry_enabled(p_grand_tour_id, p_enabled, p_changed_by, p_reason)` returns jsonb тАФ `security definer`, `service_role`-only. Flips `grand_tours.manual_result_entry_enabled`; logs to `grandtour_game_audit` (`action='admin_override'`) since it's tour-scoped, not stage-scoped.
- `public.recalculate_grandtour_stage_scores(...)` тАФ pre-existing, unmodified. `security invoker`, requires an authenticated cycling-admin session (checks `grandtour_private.is_cycling_admin()` via `auth.uid()`) тАФ **the service-role key alone cannot call this**, unlike every other RPC above.
- `public.is_current_user_cycling_admin()` returns boolean тАФ `security definer`, `set search_path = ''`. Added by `20260713010000_grandtour_is_current_user_cycling_admin_rpc.sql`. A public wrapper around the already-fixed `grandtour_private.is_cycling_admin()` (that private schema isn't exposed via PostgREST, so it couldn't be called directly). Granted to `authenticated` only тАФ Postgres grants EXECUTE to `PUBLIC` by default on a new function, so this migration explicitly `revoke all ... from public` first, then grants to `authenticated` (the same gotcha bit the local SQL test on first pass: `anon` could otherwise call it despite no explicit grant). Read-only, no writes. Used server-side by `apps/mobile/api/admin/grandtour/run-official-check.mjs` to authorize before running any check, since that route has no service-role key available to it.
- `public.correct_grandtour_stage_result_from_reviewed_report(p_stage_id, p_result_lines, p_jersey_holders, p_reconciliation, p_reason, p_request_id)` returns jsonb тАФ `security definer`. Added by `20260711010000_grandtour_correct_stage_result_rpc.sql` (Part C). Same `service_role`-or-admin-`authenticated` guard as mark-checked/finalise. **The only RPC allowed to touch an existing (draft or already-finalised/scored) result** тАФ `apply_grandtour_official_stage_result` still refuses outright if a different draft or any final result already exists; this is the deliberate, audited bypass of that refusal. Refuses a blank `p_reason`, a stage with no existing result at all (use apply instead), TTT stages, and every apply-style reconciliation/line/jersey-holder gate. On a real content change: unfinalises first (required тАФ see the trigger-ordering gotcha below), always lands on `review_status='correction_required'`/`is_final=false`, deletes any existing `grandtour_stage_scores` rows for the stage and moves affected tips from `'scored'` to `'corrected'` (never deletes/reverts them further тАФ `recalculate_grandtour_stage_scores` already keeps a `'corrected'` tip's status as `'corrected'`, not `'scored'`, after rescoring), and logs one `result_corrected` row to `grandtour_result_audit_log`. Idempotent no-op (`status='no_change'`, no writes at all) on byte-identical content. Never finalises, never scores. CLI: `scripts/grandtour-admin-stage.mjs --update-results --from-report <path> --reason <text>` (never fetches letour.fr itself тАФ same `--from-report`-only trust boundary as `--apply`). UI: `/admin/grandtour-stages`' "Update Results / Re-run Official Import" panel (paste a CLI-generated report; the UI cannot fetch letour.fr itself тАФ no server-side scraping endpoint exists in this project). Full workflow: `docs/GRANDTOUR_APPLY_MODE_PRODUCTION_READINESS_CHECKLIST.md` ┬з15.

### Postgres gotchas learned this session (apply broadly)
- Adding a parameter to a `security definer` function via a plain `create or replace function` is **not** a same-OID replace if the parameter list changes тАФ Postgres treats a different arg-type list as a distinct overload, silently leaving both old and new versions defined. Whenever a function's parameter list changes (not just its body), explicitly `drop function if exists ...(<old exact signature>)` before recreating it. `create or replace function` is only safe as a plain in-place update when the parameter list is byte-identical.
- **A `returns boolean` SQL function whose body is `select <expr> from ... where exists (...)` returns SQL `NULL` for the "false" case, not `false`** тАФ if the `where exists(...)` filters out the row, the query returns *zero rows*, and a zero-row scalar-context function call evaluates to NULL. This silently broke `grandtour_private.is_cycling_admin()`: every `if not is_cycling_admin() then raise exception ...` guard (imperative PL/pgSQL) was a no-op for non-admins, because `if not NULL` is `NULL`, and `if` only takes the branch when the condition is `TRUE`. Confirmed via direct testing that a non-admin authenticated session could call `recalculate_grandtour_stage_scores()` and sail straight past the intended admin check. **RLS `using(...)`/`with check(...)` policies were unaffected** тАФ Postgres RLS treats NULL the same as false тАФ only the `if not X then raise` pattern was exposed. Fixed in `20260710060000_grandtour_admin_check_finalise_authenticated_grants.sql` by rewriting the function body as `select exists (...)`, which always returns a genuine `true`/`false`, never NULL. **Lesson: any `returns boolean` SQL function meant to be used in an imperative `if not fn() then raise` guard must be written as `select exists(...)` (or with an explicit `coalesce(..., false)`), never as a bare `where exists(...)`-filtered row select** тАФ write a genuine negative-path test (a real non-admin session actually calling the guarded function) before trusting *any* such guard, not just a positive-path (admin-succeeds) test.
- **`grandtour_stage_result_lines`/`grandtour_stage_jersey_holders` both have "prevent final delete" triggers** (`grandtour_stage_result_lines_prevent_final_delete`/`grandtour_stage_jersey_holders_prevent_final_delete`, from `20260629080958_grandtour_mvp.sql`) that block `DELETE` while the parent `grandtour_stage_results.is_final` is true. `correct_grandtour_stage_result_from_reviewed_report` (Part C) therefore flips `is_final` to `false` **first**, in the same function call, before deleting/reinserting any line or jersey-holder row тАФ reversing that order makes correcting an already-finalised stage fail outright with an opaque trigger error. Same applies to `grandtour_stage_team_result_lines` (TTT) via `prevent_final_team_result_line_delete`, not currently exercised since TTT corrections are out of scope.

### Auth redirect URLs (sign-up confirmation, password reset)
- **Problem this fixed:** sign-up confirmation and password-reset emails were linking to `localhost` in production instead of the deployed Vercel app, because `signUpWithPassword` passed no `emailRedirectTo` at all, and the reset-password flow built its redirect with `expo-linking`'s `Linking.createURL`, which has no production-safe web behaviour guarantee.

In development (`__DEV__`), defaults to `http://localhost:8081` when unset.
- `signUpWithPassword` (`packages/supabase-client/src/auth.ts`) now always passes `emailRedirectTo: getAuthRedirectUrl("/auth/callback")`.
- `ForgotPasswordScreen.tsx` uses `getAuthRedirectUrl("/reset-password")` **only on web** (`Platform.OS === "web"`); native keeps its existing `Linking.createURL` deep link, consumed by `AuthProvider`'s native URL listener тАФ these are two genuinely different mechanisms for two different platforms, both intentional.
- **Preview deployments are deliberately NOT supported as Auth redirect targets** (chosen approach (A), not (B) тАФ see `docs/deployment.md`). `EXPO_PUBLIC_APP_URL` is set to the *same* production URL for both the Production and Preview Vercel environments, so every Auth email always lands on the one stable, allow-listed production origin. - New route `apps/mobile/app/auth/callback.tsx` тЖТ `AuthCallbackScreen.tsx`, reachable regardless of auth state (added as a `<Stack.Screen name="auth/callback" />` in `app/_layout.tsx`, outside both `Stack.Protected` blocks тАФ a sign-up confirmation must be reachable by a not-yet-authenticated user). ### Auth callback routing (fixed a stuck-spinner / stray-redirect bug on `/auth/callback`)
- **Problem this fixed:** opening the app or logging out could leave the browser sitting on `https://grandtour-three.vercel.app/auth/callback` with a permanent "Confirming your accountтАж" spinner. Root cause: `AuthCallbackScreen` never looked at whether the URL actually carried a callback code/token/error тАФ it just reflected `useAuth()`'s `loading`/`user` state and only redirected once `user` became truthy. Anyone who ever landed on that route without a live confirmation in progress (a stale tab, a bookmark, a browser-history revisit, or simply being on that route when they clicked "Log out") had no `user` and no URL error, so the screen span forever. Logout (`profile.tsx`) also never navigated anywhere after calling `signOut()`, relying entirely on the root `Stack.Protected` guard swap тАФ which only redirects *away* from screens inside a protected group, and `/auth/callback` is deliberately outside both groups (reachable regardless of auth state), so it was never swept away by that mechanism.
- **Fix:**
  - `apps/mobile/lib/authCallbackExperience.ts` (new, pure, unit-tested тАФ `apps/mobile/tests/authCallbackExperience.test.cjs`) тАФ `decideAuthCallbackAction(params)` inspects the actual callback params (`code`, `error`/`error_description`, `access_token`+`refresh_token`) and returns exactly one of `show_error` / `exchange_code` / `set_session` / `redirect_home` тАФ **`redirect_home` when none of those are present**, so "no valid callback params" always means an immediate redirect, never a spinner. `sanitizeInternalReturnPath(path)` only ever returns a same-origin relative path (rejects absolute URLs, `//protocol-relative`, `/\backslash-trick`, embedded `://` schemes, and any path that points back at `/auth/callback` itself, to prevent a redirect loop) тАФ used for an optional `?next=` return path, always falling back to `/`.
  - `AuthCallbackScreen.tsx` rewritten around that decision: on `exchange_code` it calls `supabase.auth.exchangeCodeForSession(code)` itself (PKCE, used by the sign-up confirmation link); on `set_session` it parses `window.location.hash` (`parseHashParams`) and calls `supabase.auth.setSession({access_token, refresh_token})` (implicit-flow hash tokens, if ever used); on `show_error` it stops immediately and renders the error with a "Back to sign in" button; on `redirect_home` it redirects to `/` (or the sanitized `next`) with no spinner. A 15s timeout flips any stuck in-flight exchange to an error state тАФ the screen can never spin forever, even on a hung network call. The effect runs exactly once per mount (`handledRef`), since a PKCE code and recovery hash tokens are single-use.
  - `packages/supabase-client/src/client.ts`: `detectSessionInUrl` changed from `typeof window !== "undefined"` to always `false`. This was required, not optional тАФ leaving it `true` would race Supabase's own automatic URL-based session detection against `AuthCallbackScreen`'s explicit `exchangeCodeForSession` call over the *same* one-time-use code, and whichever lost the race would surface a false "Confirmation failed" even though the user was actually signed in. The web `/reset-password` recovery-link flow does **not** depend on `detectSessionInUrl` тАФ `AuthProvider`'s `handleRecoveryUrl` already parses and applies those hash tokens independently via `Linking.getInitialURL()`/`setSession`, so it was unaffected. This app has no OAuth (`signInWithOAuth`) flow, which is the other thing `detectSessionInUrl` would normally matter for.
  - `apps/mobile/app/profile.tsx`: "Log out" now `await signOut()` then unconditionally `router.replace("/login")` in a `finally` (so it still redirects even if `signOut()` throws), instead of firing-and-forgetting `signOut()` with no navigation at all.
- **Not changed / deliberately out of scope:** `ResetPasswordScreen.tsx`'s own `signOut()` тЖТ `router.replace("/login")` after a password update was already explicit and correct; left untouched.
- Full before/after behaviour, files changed, and test/build results for this fix are in the session's chat transcript (not duplicated into HANDOFF.md's own point-in-time snapshot format beyond a summary entry).

### GrandTour auto dry-run admin notification email

### GrandTour auto dry-run: stage-resolution and exit-code fix
- **Problem this fixed:** a real scheduled run's `final-summary.json` showed `stageNumber: 1`, `finalStatus: "unsafe_review_required"` (blocked because "Stage is a TTT; official team-result source is not confirmed") тАФ and the job failed (red X) because of it, and separately the mail step failed with `Error: Server address must be specified` (see above).
- **Root cause of the stage-1 selection (not a hardcoded fallback тАФ a starvation bug):** `resolveAutomaticStage` (`scripts/grandtour-stage-calendar.mjs`) always picked the *earliest* eligible, not-yet-`isFinal` stage. A TTT stage can never become `isFinal` through this pipeline (the TTT safety rule in `scripts/grandtour-reconciliation.mjs` тАФ preserved, unchanged, still covered by its own passing test `stage 1 TTT is never safe to apply even with a perfect match` тАФ means it is never safe to apply, so it's never finalised). "Earliest eligible, not final" therefore re-selected the permanently-stuck TTT stage on *every single scheduled run forever*, even once later stages had raced and genuinely needed a first look. No literal `|| 1` fallback existed anywhere in the code тАФ the selection logic itself produced `1` correctly, every day, by design, which was the actual bug.
- **Fix:** `resolveAutomaticStage` now prefers the **latest** eligible stage (`resolutionSource: "database_schedule"`); only if that latest one is already finalised (and `allowRerunCompleted` is false) does it fall back to the most recent still-unresolved older stage instead (`resolutionSource: "unresolved_stage"`) тАФ so a genuine straggler is still self-healed, but a structurally-unresolvable early stage (like a TTT) can never again starve out attention from every later stage. Returns `resolutionSource: "none"` (never a hardcoded stage) when nothing is eligible. `resolveStageRange` (`scripts/grandtour-auto-dry-run.mjs`) adds a fourth source, `"manual_input"`, for explicit `--stage-number`/`--from-stage`/`--to-stage`, and now logs the resolution source, selected stage number, its start time, the current time, and the reason on every call.
- **Exit-code fix:** `computeExitCode` no longer treats `unsafe_review_required` as failure-by-default. A completed dry run with `finalError: null` and `finalStatus: "unsafe_review_required"` now **always** exits 0 (a `::warning title=GrandTour review required::<blockers>` annotation is printed instead) тАФ it produced a real report; there's nothing to retry or fix in CI, only a human review to do. Exit 1 is reserved for genuine technical failures: any non-null `finalError`, plus `parser_drift`/`transient_failure_exhausted`/`configuration_error`/`unexpected_failure`. The `--fail-on-unsafe` CLI flag/workflow input is kept (parsed, still accepted) for backward compatibility but is now inert тАФ it no longer affects the exit code.
- **Not changed:** the TTT safety rule itself (`scripts/grandtour-reconciliation.mjs`) тАФ a TTT is still never safe to apply until its official team-result source is confirmed, regardless of any of the above.
- Tests: `scripts/grandtour-stage-calendar.test.mjs` and `scripts/grandtour-auto-dry-run.test.mjs` cover the latest-first/straggler-fallback/none resolution paths (including a dedicated "never defaults to stage 1" case), the unconditional `unsafe_review_required` тЖТ exit 0 behaviour, the GitHub warning annotation, and a genuine `finalError` still forcing exit 1; `scripts/grandtour-auto-dry-run-workflow.test.mjs` covers the new SMTP env-var mapping and both mail-step/skip-notice guards.

## Working conventions in this repo
- Local Supabase: `npx supabase status` for connection details; local demo anon/service-role keys are well-known Supabase CLI defaults, not secrets.
- Always rehearse against **local Supabase only** before any production action; production actions require explicit user authorization in the task itself, and even then double-check any pasted key's `role` claim (JWT) before using it тАФ don't assume a pasted "anon key" or "production key" is actually what it's labeled.
- `npm run test:data` runs all `scripts/*.test.mjs` via Node's built-in test runner. `supabase/tests/*.sql` are pgTAP-style scripts run via `docker exec ... psql -f`, each in its own `begin`/`rollback` transaction. `npm run grandtour:reconcile:local-smoke`, `npm run grandtour:apply:local-smoke`, and `npm run grandtour:admin-stage:local-smoke` are real-Supabase (local) end-to-end smoke tests, not mocked тАФ the apply and admin-stage ones perform real writes and clean up after themselves (including any test `auth.users` rows they create via `signUp`). The admin-stage smoke test's `--grand-tour-name`/`--grand-tour-year` fixture values must match `supabase/seed.sql`'s local grand tour (`"GrandTour France 2026"`, 2026) тАФ the CLI's own real-world default (`"Tour de France"`) is a production concept, not a local one. From `apps/mobile`: `npm run test:ui` (pure UI logic, compiled via `tsc` into `dist/mobile-tests` then run with `node --test`) and `npm run test:api` (the `apps/mobile/api/**/*.mjs` server routes, `node --test` directly тАФ no `tsc` step, since `apps/mobile/tsconfig.json`'s `include` deliberately excludes `api/`).
- Root `npm test` now runs `packages/tipping-core`'s test **and** `packages/supabase-client`'s test (`node --test src/*.test.ts`) in sequence. `packages/supabase-client` has no build step тАФ Node 24 runs `.test.ts` files directly via its native TypeScript stripping (unflagged since Node 23.6+), as long as intra-package imports use an explicit `.ts` extension (Node's ESM resolver requires it; `tsc`/bundler resolution doesn't need it and would reject it without `allowImportingTsExtensions`, hence root `tsconfig.json` now `exclude`s `**/*.test.ts` from the main typecheck rather than enabling that compiler option repo-wide).
- **Node-in-CI gotcha:** `@supabase/supabase-js`'s `createClient()` throws synchronously on Node 20 (`Node.js 20 detected without native WebSocket support`) because its Realtime client requires a WebSocket constructor and no `ws` package is installed as a fallback in this repo. Any GitHub Actions workflow that constructs a Supabase client (i.e. runs `--reconcile`, or anything else that calls `createClient`) needs `node-version: '22'` or higher тАФ `grandtour-auto-dry-run.yml` hit this on its first real run (fixed by bumping `'20'` тЖТ `'22'` тЖТ `'24'`, alongside `actions/checkout@v5`/`actions/setup-node@v5`). `grandtour-daily-feed-dry-run.yml` never calls `--reconcile` so it was never exposed to this, but keep it in mind for any future workflow that does.
- **Postgres gotcha:** a newly-created function gets EXECUTE granted to `PUBLIC` by default. `grant execute on function ... to authenticated` alone does **not** revoke that тАФ `anon` (or anyone) can still call it. Any function meant to be admin/authenticated-only needs an explicit `revoke all on function ... from public;` before the intended grant (see `20260713010000_grandtour_is_current_user_cycling_admin_rpc.sql`, and the SQL test that actually caught this by exercising the `anon` role, not just `authenticated`).
- **PL/pgSQL gotcha for test authors:** an `exception when others` block implicitly rolls back to a savepoint taken at the *start* of that block тАФ so a deliberate `DELETE`/setup statement meant to persist past the exception must be a separate top-level statement, never inside the same `do $$ ... exception ... $$` block being tested.
