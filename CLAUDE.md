## Dashboard, stage closure state, and result eligibility

### Stage closure state (open/closing_soon/closed/live/completed)
- `packages/tipping-core/src/cycling-stage-tip.ts` exports `resolveCyclingStageClosureState(input)` — the single source of truth for whether a stage is currently open for tipping. Previously the mobile app hand-rolled `now >= stage.locks_at` three different ways (dashboard, `LockCountdownCard`, stages list), each with slightly different null-handling and **none of them read `grandtour_stages.manual_locked_at`** (an admin override) at all. This function fixes both: it extends `CyclingStageLockInput`/`resolveCyclingStageLockAt`/`isCyclingStageTipLocked` with `locksAt`/`manualLockedAt` fields (priority `manualLockedAt > locksAt > startTime(legacy) > stageDate+defaultLockTimeUtc`, additive/backward-compatible), then layers `live` (now >= `starts_at`) and `completed` (`isFinal === true`) around that boolean, in that priority order (`completed` > `live` > the lock-derived `closed`/`closing_soon`/`open`). `closing_soon` is <24h before lock. A single `now` is resolved once per call and reused throughout — callers must do the same (resolve `now = new Date()` once per render, never call `new Date()` repeatedly while building one card).
- `apps/mobile/lib/stageClosureExperience.ts`'s `buildClosureDisplay(state, ...)` turns that resolved state into exactly what a card renders: badge label, primary text line, emphasis (true for live, and for closing_soon under 60 minutes), whether tip-editing is allowed, and the CTA label. Deliberately takes the state as a plain string union (not a runtime import of the tipping-core type) and a pre-formatted date/time string — this file is compiled standalone by `apps/mobile/package.json`'s `test:ui` script (see below), which cannot resolve cross-package runtime imports.
- `apps/mobile/components/StageStatusBadge.tsx` — the compact Open/Closing soon/Closed/Live/Provisional/Completed pill used across the dashboard. Visually redesigned (see below) to a single neutral chip + the one accent colour for emphasis only - it no longer gives each tone its own background hue.
- `apps/mobile/app/stages/index.tsx` calls `resolveCyclingStageClosureState` instead of its own inline date math (now also fetching `useCyclingStageResults` so it can genuinely reach `completed`, not just fall through to `live` for any past-started stage — see "GWFC UI correction pass" below). `apps/mobile/app/stages/[stageId].tsx` (the actual tip-entry screen) was originally **deliberately left untouched**, and its `preselection`-mode lock logic still is (a genuinely different, race-level lock concept with no equivalent shared resolver) — but its `daily`-mode lock now also goes through `resolveCyclingStageClosureState`, added in the same later pass; see below for why and what changed. `LockCountdownCard.tsx` was removed entirely in the visual redesign below - its content is now folded into the dashboard's single action card, and nothing else used it.

### Stage-result eligibility ("future stages must never appear as results")
- `packages/tipping-core/src/stage-eligibility.ts` exports `isStageEligibleForResults(stage, now)` and `selectLatestEligibleStage(candidates, now)`. The real schema has no `stage.status`/`result.result_status` columns matching a generic in_progress/completed/provisional/official vocabulary — this adapts the conceptual rule to what actually exists: `stage.starts_at <= now AND (result.is_final === true OR result.review_status is one of imported/review_required/admin_checked/correction_required)`. A stage row existing (or even a `draft` result existing) is never treated as a real result on its own.
- **Real constraint, not a gap to "fix" later:** the public app's anon/publishable key can only ever see `grandtour_stage_results` rows where `is_final = true` (RLS — see the GrandTour pipeline section above). So `reviewStatus`/"provisional" support in the predicate exists for correctness and any future/admin-authenticated caller, but the public dashboard today will only ever pass `isFinal`, never a genuine `provisional` result — this is expected, not a bug.
- Applied at three layers, per the task's "earliest reliable layer" requirement: (1) `packages/supabase-client/src/cycling.ts`'s `listCyclingStageResults`/`getCyclingStageResult` now also filter `.lte("grandtour_stages.starts_at", now)` at the query layer (both took a new optional `now: Date = new Date()` param, backward compatible); (2) the shared `isStageEligibleForResults`/`selectLatestEligibleStage` selectors, called directly from `apps/mobile/app/index.tsx` and `apps/mobile/app/results.tsx`; (3) a final defensive filter in those same screens. "Latest completed stage" is always computed via `selectLatestEligibleStage` (sorted by actual `starts_at` descending, stage number as a deterministic tie-breaker) — never an unordered client-side `.find`/re-sort-by-stage_number, which is what both screens did independently before (and which had no time-based guard at all).

### Dashboard (`apps/mobile/app/index.tsx`) — original information-hierarchy pass
Restructured into a fixed hierarchy: (1/2) a single hero card that shows either "action required" (soonest stage still open/closing_soon for tipping) or, if nothing is open, a currently-live stage — merged into one card rather than two, since in this one-stage-per-day model they're normally the same stage in different states; (3) latest eligible result; (4) user competition summary (rank/behind-leader/last-stage stats, jersey holders, mini leaderboard); (5) upcoming stages (excluding whatever's already shown as the hero); (6) recent results (excluding the stage already shown in position 3). Added loading skeletons and error/empty states — the dashboard previously had none at all. **Superseded by the visual redesign below**, which kept all of this data logic but changed the presentation hierarchy again (user-centric, not event-centric).

### Reusable clickable card
`apps/mobile/components/InfoCard.tsx` (the shared card chrome nearly every screen already used) gained optional `href`/`onPress`/`accessibilityLabel`/`accessibilityHint`/`chevron` props rather than a new parallel component. `href` renders via `expo-router`'s `Link` with `asChild` (real `<a href>` semantics, keyboard focus, and cmd/ctrl-click-to-open-in-new-tab on web), wrapping a `Pressable` with `accessibilityRole="button"` and a pressed-state opacity change; `chevron` defaults to on whenever the card is interactive. `JerseyHolderCard.tsx` got the same optional `href` treatment directly (kept as its own lightweight component rather than rebuilt on `InfoCard`, since its internal layout genuinely differs). Both are fully backward compatible — every other existing call site across the app (which never passes `href`/`onPress`) renders identically to before. `apps/mobile/lib/stageClosureExperience.ts` also exports small pure `build*DashboardCardLink` functions (href + accessibility label/hint per card type) so navigation targets and labels are unit-tested the same way the rest of this app's view-model logic is (no React rendering test harness exists in this repo, and none was added — see Testing conventions below).

**Gotcha found by actually rendering this in a browser (see below): `<Link asChild href={...}><Pressable style={({pressed}) => [{flexDirection: "row", ...}, ...]}>` collapses to vertical/stacked layout on web.** When `Link`'s `asChild` clones a `Pressable` onto a real `<a>` element, that anchor does not reliably inherit React Native Web's implicit `display: flex` the way a plain `View`/`Pressable`-rendered `<div>` does. The fix, used throughout: never put `flexDirection`/`alignItems`/`justifyContent` directly on the style passed to a `Link`-wrapped `Pressable` - put it on a plain inner `<View>` that is the Pressable's child instead, and keep the Pressable/anchor's own style to non-flex concerns (padding, minHeight, opacity-on-press). `InfoCard.tsx` already did this by accident (its row layout was always on an inner `headerRow` View); `JerseyHolderCard.tsx` and the dashboard's own link rows did not, and silently rendered broken (stacked) on a real page load despite passing `tsc`/unit tests, which only exercise the pure logic layer, not actual rendered layout.

Keeping the names stable means every other screen in the app (profile, leaderboard, stages list, results, admin) inherited the calmer palette automatically, with zero code changes there - only the dashboard itself and its directly-shared components (`InfoCard`, `StageStatusBadge`, `TipStatusBadge`, `StageTypeBadge`, `JerseyHolderCard`, `AppShell`, `DataState`) got structural/density changes. `apps/mobile/config/grandtour.ts`'s `theme.*Color` values were updated to match (used only by `AppShell`'s header background and active-nav-item colour).
- Jersey colours (yellow/green/polka-dot/white) were kept, deliberately - they're real competition information, not decorative chrome - but shrunk from a full coloured card to a single small dot next to the rider's name.
- `DashboardStatCard.tsx` and `LockCountdownCard.tsx` were deleted (dead code - the dashboard no longer uses a 3-up stat-card grid or a separate countdown card; nothing else referenced either).
- **Verification**: real headless-browser testing (Playwright, `chromium`, installed throwaway into the scratchpad - never added to this repo's `package.json`), not just `tsc`/unit tests, against a real signed-in session on **local** Supabase (an admin-API-created throwaway test user, deleted afterward - never touches production auth/data). - **New `apps/mobile/lib/leaderboardExperience.ts`** (`buildLeaderboardDisplayItems`, added to `test:ui`'s compile list + 6 new tests): when the signed-in user is outside the visible top block, shows the top N rows, a `⋯` divider, then a small window centred on the user - never forces scrolling through the whole field to find yourself. Falls back to showing every row flat (no divider) when the list is short or the user is already near the top.
- **Real bug found and fixed while visually verifying with real data (not the throwaway-empty-state check the prior redesign's own next-steps list had flagged as still-outstanding): `InfoCard`'s `accent` hero card silently rendered with no background colour at all** when wrapped by `Link asChild` (the dashboard's "what should I do next" action card) - text/badge/button all rendered, but transparent instead of on a solid brand-blue fill, making it barely legible against the page background. Root cause: the card's visual styling was on the Link-wrapped `Pressable`'s own `style` prop, passed as a **function** (`({pressed}) => [...]`, so Pressable could vary it by press state) - expo-router's `<Link asChild>` clones its child's props onto a real `<a>` on web, and a function-valued `style` does not survive that clone; the resulting anchor silently got no style at all, not just no pressed-state variant. This is a different/worse failure mode than the earlier documented `flexDirection`-only collapse bug in this same file's history. Fixed by moving all visual styling (background/border/padding/shadow, not just flex) onto a plain inner `View`, and tracking pressed-state via local `useState` + `onPressIn`/`onPressOut` instead of Pressable's render-prop style function - so the Pressable/anchor's own `style` prop is now always a plain, clone-safe value. **Lesson, extending the earlier one**: on this Link/Pressable/`asChild` combination, don't just avoid flex properties on the outer element - avoid a *function*-valued `style` prop on it entirely.
- **Real, pre-existing bug found (not caused by this session, not fixed in code) while chasing why a signed-in test user's dashboard showed no display name**: `20260701081334_canonical_grandtour_tipping_rpcs_rls.sql` does `revoke all privileges on table public.profiles from authenticated;` and then grants `SELECT` on a *different* table (`grandtour_league_profiles`) - it never re-grants `select on public.profiles to authenticated`, silently undoing the correct grant an earlier migration (`20260701053811_harden_auth_table_privileges.sql`) had established. Every authenticated user's own-profile read (`display_name`, `avatar_url`) has been failing with a genuine Postgres permission-denied (not just an empty RLS-filtered result) ever since - masked in the app by a silent "there"/blank-name fallback, so no error was ever visible to a user. **This affects production too** (that migration predates the last-verified production migration boundary). Fixed *locally only* (a direct `grant select on public.profiles to authenticated;`, not a migration - reverted by the next `supabase db reset`) purely to unblock this session's own visual verification. **Now fixed for real** in a later session — see "`public.profiles` SELECT grant — fixed for real" further down for the actual migration and production push.
- **Verification**: same real-headless-browser convention as the original redesign, this time actually exercising populated data (not just the empty-state fallback) - had to work through a chain of local-only blockers to get there (the `profiles` grant bug above; `grand_tours.source_url` being `NULL` locally, which `getCyclingRaceByYear` requires non-null, so the whole race→competition→leaderboard hook chain silently returned nothing; the post-lock tip-visibility RLS rule correctly hiding other players' picks for a not-yet-locked stage, worked around for this one screenshot by temporarily granting the verification account cycling-admin). All throwaway data, the temporary admin grant, and the temporary `source_url` edit were deleted and the local DB was reset to a clean state afterward.

## TDF 2026 rider importer

### Purpose
A standalone CLI (`scripts/tdf-2026-rider-importer.mjs`) that builds a full 184-rider TDF 2026 roster CSV/JSON, enriched with date of birth, nationality, current team, and team history, for review before being applied to `public.grandtour_riders`. Independent of the official-letour **results** pipeline documented below (different data: startlist/biographical, not stage results) — the two share no code except the generic Supabase-apply safety conventions (service-role-key gating, production URL refusal, dry-run default). Built across two sessions: an initial version replacing a broken CyclingFantasy-based importer, then a full revision replacing ProCyclingStats (also 403-blocked) with UCI's public rider-details data as the primary enrichment source.

### Source hierarchy (current)
1. **`https://www.letour.fr/en/riders`** (official Tour de France site) — bib number, rider name, team, race nationality. Reuses the already-verified `parseOfficialTourRidersHtml` from `scripts/letour-official-riders.mjs` (same parser the official-letour **results** pipeline uses for jersey/team data) rather than duplicating it. This page's markup has no rider-status (DNF/OTL/etc.) field at all — confirmed by inspecting a real cached fetch — so `status` is always written as `null` from this source (merge rules never let that overwrite an existing status).
2. **UCI's public rider-details/search data surface** — UCI rider id, canonical name, date of birth, nationality, current team, full team history. See "UCI data surface" below for exactly what this is and how it was found.
3. **Existing Supabase `grandtour_riders` row** (if any) — preserves a valid existing DOB, specialty, and any other non-null field; a UCI-sourced value only overwrites when it's genuinely present and (for DOB specifically) trusted enough — see "Merge and confidence rules" below.

**CyclingFantasy and ProCyclingStats are both removed from this pipeline entirely** — both returned HTTP 403 to every automated fetch attempt across two sessions. No workaround (different user agent, headless-browser rendering, proxying, CAPTCHA-solving, etc.) was ever attempted for either, per this project's standing safety rule against bypassing access restrictions. **Wikidata was considered and explicitly rejected** as the primary source once UCI's own data surface was confirmed reachable and richer (UCI is the actual governing body's own rider-of-record data, not a third-party aggregation) — Wikidata is not used anywhere in this importer, not even as a fallback (no session has needed one; a fallback path was never built, since it wasn't required and this project avoids building unrequested speculative code paths).

### UCI data surface — how it was found and what it is
Investigated via robots.txt (`Allow: /`, no restrictions), static HTML/bundle inspection, and — once those hit the limits of what a plain `curl` can reveal about a client-side-rendered page — a real headless-browser session (Playwright, the same throwaway-install convention already used elsewhere in this project for "what does this page's own JS actually request" problems, see "Auth callback routing, part 3" above) loading `uci.org`'s Road Riders/Teams page and capturing its own outgoing network requests.

- **The mechanism**: `GET https://www.uci.org/api/riders/<disciplineCode>/<year>?page=<n>&search=<query>` — a genuine first-party, unauthenticated, paginated JSON API backing UCI's own visible rider list/search box. Confirmed live with a plain `curl` (200 OK, no cookies, no auth, no CAPTCHA/Cloudflare challenge). `disciplineCode="ROA"` for road. `search` performs a case-insensitive **substring** match across `givenName`/`familyName` combined — but is **not accent-folding** (`search=Pogacar` misses UCI's own `"POGAČAR"`; `search=Poga`, the accent-free prefix, hits). This is why `buildUciSearchQueries` tries the whole official name first, then falls back to individual name words: letour.fr's own rendered names are frequently ASCII-only (confirmed live: the real letour page renders `"TADEJ POGACAR"`, no accent) even when UCI's own record has the correct diacritic.
- A separate Algolia-powered site-search widget was found first (`data-component="SearchOverlay"`, index `uci-search-en-gb-master`) and **ruled out** — its own facet counts confirm it only indexes CMS content (page/news/event/resource), never rider records.
- **Rider profile pages** (`https://www.uci.org/rider-details/<id>`) are server-rendered HTML with a clean embedded JSON payload in `<div data-component="RiderDetailsModule" data-props="{...}">` — `{details: {givenName, familyName, location (current team), dob ("DD.MM.YYYY"), nationality, sanctions}, history: {teams: [{teamName, teamCode, countryCode, url, year}]}}`. No numeric rider id can be derived from a name — it's an opaque UCI-assigned id, only discoverable via the search API (or, for investigation purposes only, a plain web search — never used as part of the importer's own production code path).
- **Parsing prefers this structured JSON over rendered-text scraping throughout** (`scripts/tdf-2026-uci-parsers.mjs`'s `extractDataComponentProps` extracts and `JSON.parse`s the bounded `data-props` attribute value directly) — this is a deliberate reaction to a real bug found and fixed *while writing this session's own tests*: an earlier draft of the field extractors searched a fully tag-flattened version of the page for label text, and once every HTML tag is collapsed to a single space, two adjacent label/value pairs (e.g. "Date of birth: ... Nationality: ...") become indistinguishable — a lazy "match up to the next colon" regex has no way to know where one field's value ends and the next label begins, so one field's value could bleed into another's. Fixed before it ever shipped, but the same failure mode is exactly what motivated moving off PCS's rendered-text parsing in the first place (see "Removed sources" below) — worth remembering for *any* future HTML-scraping parser in this codebase: never flatten-then-regex when values must stay isolated to their own field; parse structured JSON if it's present, or work on bounded raw-HTML windows if it isn't.

### Merge and confidence rules
- **Name/nationality/team confidence scoring** (`scripts/tdf-2026-uci-match.mjs`'s `scoreUciCandidate`): deterministic tiers, never a fuzzy score threshold.
  - **High**: name matches (exact, or alternate given/family order) **and** nationality agrees **and** current team agrees (UCI's team name always has a trailing `(CODE)` suffix — `stripUciTeamCodeSuffix` strips it before comparing).
  - **Medium**: name matches **and** nationality agrees, but team is missing or differs (naming-convention drift, an untracked transfer, etc).
  - **Low**: anything else — no name match at all (a fuzzy/partial hit only, e.g. UCI's fuller `"DEL TORO ROMERO"` vs letour's `"Del Toro"` — a real case seen in this session's own live run, correctly kept conservative rather than guessed), a confirmed nationality conflict, or a team name suggesting a different discipline (a small keyword denylist: TRACK/BMX/MOUNTAIN BIKE/MTB/CYCLO-CROSS/INDOOR CYCLING).
  - `pickBestUciMatch` never silently picks the first of several plausible (high/medium) candidates — if more than one candidate independently scores high/medium, the whole match degrades to low-confidence/"multiple plausible candidates" rather than guessing. A lone low-confidence candidate is still surfaced (for the review file), never discarded outright.
- **`mergeDateOfBirth`** (`scripts/tdf-2026-rider-match.mjs`) is DOB's own merge rule, stricter than every other field's generic "incoming wins if present" rule (`mergeRiderField`): **only a high- or medium-confidence UCI match may populate an empty DOB**; a low-confidence match never writes DOB even when there's nothing there yet. When both an existing (trusted) DOB and a new, sufficiently-confident incoming DOB are present and **differ**, the existing value is kept and a `conflict: true` flag is returned (surfaced in the review CSV and the `dobConflicts` summary count) — this is a real behavioural change from the generic field-merge rule, not an oversight; every other writable field still uses the plain null-safe "incoming wins" rule.
- **Specialty is never freshly computed by this importer.** UCI's rider-details payload has no specialty/discipline-score field at all (confirmed by inspecting the real payload — just name/DOB/nationality/team/team-history). `resolveSpecialty` (`scripts/tdf-2026-rider-specialty.mjs`) only ever preserves whatever's already in the matched Supabase row's `specialities` array (mapped display-only onto this importer's ten-value output vocabulary — `leadout`/`breakaway` have no equivalent and display as `"unknown"`, but the underlying DB value is never touched since this importer never writes a fresh `specialities` value), or reports `"unknown"` with `specialty_source: "unknown"` if there's nothing to preserve. `specialty_source: "manual"` is a defined-but-currently-unreachable value — there is no manual-review/provenance column on `grandtour_riders` to distinguish a manually-reviewed value from any other pre-existing one; a disclosed schema gap, not a bug in this importer. (An earlier version of this importer, when ProCyclingStats was still source #2, *did* compute a fresh specialty from PCS's "points per specialty" scores with documented deterministic thresholds — that mechanism was removed wholesale along with PCS itself, not adapted, since UCI has no equivalent signal.)
- **Young-rider (white jersey) eligibility is unchanged across both sessions**: `youngRiderEligibilityCutoffDate(raceYear)` = `${raceYear - 25}-01-01` (a fixed calendar cutoff derived only from the race year, never "today"'s date) — `2001-01-01` for the 2026 Tour, boundary inclusive. Sourced to "UCI/ASO young rider (white jersey) classification: riders aged 25 or younger during the calendar year of the race." Verified already documented/tested from the first session and retained as-is in the UCI revision, per that revision's own explicit instruction to re-verify rather than blindly re-derive it.

### Circuit breaker
`createCircuitBreaker(threshold = 3)` (`scripts/tdf-2026-rider-fetch.mjs`) opens after **3 consecutive** 403/429 responses from a source (a 5xx, timeout, or success in between resets the streak — the breaker exists to detect "this source has started actively blocking us," not to penalize occasional transient failures). Once open, it **stays open for the rest of that process's run** (no half-open retry probing within one run — a fresh process starts with a fresh, closed breaker). Wired into the UCI client only (`scripts/tdf-2026-uci-client.mjs`): once open, `discoverUciCandidates`/`fetchUciRiderProfile` skip the network entirely (`UciCircuitBreakerOpenError`, thrown with zero further requests) for every remaining rider in the run — the importer's main loop checks `circuitBreaker.isOpen()` before even attempting a rider's UCI lookup, so a tripped breaker produces **exactly one** aggregate `sourceFailures` entry after the loop (never one row per skipped rider), while every remaining rider still gets a full roster entry from letour.fr (source #1 is never gated by UCI's health). Verified live this session: 0 activations across the real 184-rider run (UCI never blocked this session at all).

## Master UCI Rider Registry & Weekly Sync

### Purpose
A cross-race canonical rider identity registry (`public.uci_riders` + satellite tables), sourced primarily from UCI's public rider data surface, that any race's start list can be matched against — without disturbing `public.grandtour_riders`' existing tour-scoped schema, RLS, or the hundreds of tests that already depend on it. Generalizes the TDF 2026 rider importer's proven UCI access code (`scripts/uci-client.mjs`/`uci-parsers.mjs`/`uci-match.mjs`/`source-fetch-utils.mjs`, all renamed from their `tdf-2026-`-prefixed originals — see above) into a standalone weekly-sync pipeline. Built in one session; local-only so far (no migration or sync has been pushed/run against production — see "Not done this session" below).

### Why not just extend `grandtour_riders`
`grandtour_riders`/`grandtour_teams`/`grandtour_stage_startlists` are all tour-scoped (`grand_tour_id`/`stage_id not null`, cascade-tied) and sit at the center of an FK graph tuned for a single tour's lifecycle (`grandtour_tip_selections`, `grandtour_stage_result_lines`, `grandtour_stage_jersey_holders`, `grandtour_favourite_riders` all point at `grandtour_riders.id`). Repurposing it as a cross-race master table was explicitly rejected — it would either break that FK graph or force every other race's rider to squeeze into one tour's row shape. `grandtour_riders` also has no external-id column at all (`source_url` is used as an ad hoc stable key by convention only). The registry instead **links, rather than replaces**: a single nullable `grandtour_riders.master_rider_id uuid references uci_riders(id) on delete set null` column (added in `20260717050000_grandtour_riders_master_rider_id.sql`) is the only change to the existing schema — every existing FK, RLS policy, and downstream consumer of `grandtour_riders` is untouched. Team linkage is deliberately **not** a foreign key on the canonical rider either: `grandtour_teams` is itself tour-scoped, so a rider's UCI-sourced current team and team history are stored as source text (`current_team_name`/`current_team_code`, `uci_rider_team_history.source_team_name`/`source_team_code`) — resolving a specific `grandtour_teams.id` happens later, contextually, only during race-entry matching for one particular race, on exact normalized name/code equality, never by fuzzy similarity.

### Schema (4 migrations, `20260717020000`–`20260717050000`, local-only)
- **`public.uci_riders`** (`20260717020000_uci_rider_registry_schema.sql`) — the canonical registry. `id`, `uci_rider_id` (unique via a partial index where not null — the strong, opaque, UCI-assigned identity signal), `uci_code`, `given_name`, `family_name`, `display_name not null`, `normalized_name not null` (plain trim/lowercase/collapse — matches `dbNormalizedName`'s convention, **not** accent-folded, and carries no unique constraint of its own — names are never a sole identity key here), `date_of_birth`, `nationality`, `gender`, `discipline not null default 'road'`, `current_team_name`, `current_team_code`, `uci_profile_url`, `is_active not null default true`, `source_updated_at`, `last_verified_at`, `last_seen_at`, `consecutive_absences not null default 0`, `data_confidence` (low/medium/high), `manual_review_required not null default false`, timestamps. RLS: public `select` (`anon`+`authenticated`); all writes `service_role`-only (no authenticated write path — matches `grandtour_riders`/`grandtour_teams`' own convention; the sync CLI, not the app, is the only writer).
- **`public.uci_rider_aliases`** (`20260717030000_uci_rider_satellite_tables.sql`) — alternate names for race-entry matching. `rider_id → uci_riders cascade`, `alias_text`, `normalized_alias`, `alias_type` (new enum `uci_rider_alias_type`: `uci_canonical, surname_first, given_name_first, accentless, race_organiser, abbreviated, former_name, manual`), `source`, `confidence`. Unique on `(rider_id, normalized_alias, alias_type)` — **deliberately not** globally unique on `normalized_alias` alone, since shared names across different riders are legal and expected. Same public-read/service-write RLS shape.
- **`public.uci_rider_team_history`** (same migration) — per-season team history from UCI's `teamHistoryRaw`. `rider_id → uci_riders cascade`, `team_id → grandtour_teams set null` (resolved opportunistically, see below), `source_team_name not null`, `source_team_code`, `season_year`, `valid_from`/`valid_to`, `discipline default 'road'`, `source default 'uci'`. Idempotent upsert key is a **unique index on an expression** — `(rider_id, season_year, coalesce(source_team_code, ''), source)` — not a plain table `unique(...)` constraint, since Postgres doesn't accept a function call like `coalesce()` in a bare column-list constraint; this also means the sync layer can't rely on PostgREST/`supabase-js`'s `upsert({onConflict: "..."})` (which only accepts a column list, not an expression) — `scripts/uci-rider-sync-supabase.mjs` instead reads existing rows, computes the same key in JS, and issues an explicit insert-or-update itself, matching this expression-index constraint by construction.
- **`public.uci_rider_specialties`** (same migration) — season-aware, kept deliberately separate from core identity (`uci_riders` itself carries no specialty column at all) because UCI's data surface supplies no specialty signal whatsoever. `rider_id → uci_riders cascade`, `season not null`, `primary_specialty`/`secondary_specialty` (checked against the *existing* ten-value `SUPPORTED_SPECIALTIES` vocabulary from `scripts/tdf-2026-rider-specialty.mjs`, reused not redefined), `confidence`, `evidence jsonb`, `source` (existing_supabase/manual/unknown), `manually_reviewed`, `reviewed_at`. Unique `(rider_id, season)`.
- **`public.uci_rider_review_queue`** + **`public.uci_rider_sync_runs`** (`20260717040000_uci_rider_review_queue_and_sync_runs.sql`) — same admin-read/service-write RLS shape already used by `grandtour_feed_import_runs`/`grandtour_feed_snapshots` (select restricted to `grandtour_private.is_cycling_admin()`). The review queue's `queue_type` enum (`unmatched_startlist_rider, ambiguous_candidate, dob_conflict, nationality_conflict, team_mismatch, duplicate_uci_identity, suspected_duplicate_internal_rider, low_confidence_alias_match`) and `status` enum (`pending, matched, new_rider_approved, source_correction, ignored, resolved`) capture everything the sync/matching pipeline can't resolve automatically — never silently dropped. `uci_rider_sync_runs` is the audit trail for every sync invocation (`mode`: dry_run/apply; `status`: running/completed/failed/partial), mirroring `grandtour_feed_import_runs`' role for the results pipeline.
- **`public.resolve_uci_rider_review_item(p_item_id, p_status, p_resolved_by, p_note, p_create_alias default null)`** (same migration) — `security definer`, same `auth.role() = 'service_role' or grandtour_private.is_cycling_admin()` guard pattern as `mark_grandtour_stage_result_checked`/`finalize_grandtour_stage_result`. Updates the queue row and, when `p_create_alias` (`{rider_id, alias_text, alias_type}`) is supplied, inserts a new `uci_rider_aliases` row in the same call — the "an admin approving a race-name match teaches the registry that alias for next time" requirement. Granted to `authenticated` + `service_role` (explicit `revoke all` first, per this repo's standing Postgres grant-gap gotcha).
- **`grandtour_riders.master_rider_id`** (`20260717050000_grandtour_riders_master_rider_id.sql`) — the one link column described above. Nothing else about `grandtour_riders` changes.
- `packages/shared-types/src/database.ts` regenerated (`npx supabase gen types typescript --local`) after `db reset` — diff confirmed to be exactly the four new tables/enums plus the one new column, nothing else.

### Script layer (`scripts/`)
- **`uci-rider-registry.mjs`** — canonical-rider merge/match logic against `uci_riders` (not `grandtour_riders`). `matchCanonicalRider` — priority `uci_rider_id` (authoritative) then `normalized_name` (fallback, flagged non-authoritative; >1 candidate at either tier is reported, never guessed). `mergeTrustedField`/`mergeCanonicalRiderRecord` generalize `tdf-2026-rider-match.mjs`'s DOB-only conflict rule to **both** `date_of_birth` and `nationality`: a manually-reviewed or higher-confidence existing value is never silently overwritten by null/lower-confidence incoming data; a genuine conflict keeps the trusted existing value and reports `conflict: true`. `sourceContentHash`/`planRegistrySync` compute an unchanged-vs-changed decision by comparing a freshly-merged candidate's own field hash against the existing row's own fields reshaped into the same key order — **not** a stored hash column (none exists on `uci_riders`) — so an identical resync writes nothing.
- **`uci-rider-aliases.mjs`** — deterministic alias generation (`uci_canonical`, `surname_first`, `given_name_first`, `accentless` via the existing `normalizeRiderName` accent-fold, `abbreviated`) with an explicit rejection rule for unsafe bare-initials/below-minimum-signal-length aliases (`isUnsafeBareInitials`, `MIN_ALIAS_SIGNAL_LENGTH = 4`) — an alias that thin would match far too many unrelated riders during race-entry matching. `planRiderAliasSync` plans inserts keyed on `(rider_id, normalized_alias, alias_type)`; an existing alias is never updated in place (a manual alias's higher confidence must never be silently downgraded by a later deterministic re-run).
- **`uci-rider-team-history.mjs`** — turns UCI's `teamHistoryRaw` into planned upserts. `resolveTeamId`/`buildTeamLookupIndex` only ever resolve a `grandtour_teams.id` on **exact** normalized name/code equality (reusing `normalizeTeamName`) — never on fuzzy similarity, so a naming-convention drift year-to-year (a real, observed case) correctly stays unresolved rather than being auto-merged onto the wrong team.
- **`uci-rider-specialty.mjs`** — imports (does not duplicate) `SUPPORTED_SPECIALTIES`/`resolveSpecialty` from `tdf-2026-rider-specialty.mjs`. Never infers a fresh specialty from UCI data (UCI has none); only ever preserves a trusted existing value or reports `unknown`. A `manually_reviewed` existing row is always preserved outright, regardless of what a resync would otherwise resolve.
- **`uci-rider-inactivity-policy.mjs`** — pure absence policy. `last_seen_at` refreshes whenever a **completed** sync finds the rider; `consecutive_absences` increments only on a completed run where the rider was expected but absent (a `partial`/`failed` run is never evidence of absence — never touches the counter at all); `is_active` flips to `false` only once `consecutive_absences` reaches `INACTIVITY_THRESHOLD_CONSECUTIVE_SYNCS` (4, ≈1 month of weekly syncs — a count of consecutive completed runs, not a calendar duration). Never a hard delete; reappearing in any later completed sync reactivates the rider automatically.
- **`race-entry-rider-matching.mjs`** — the generalized, race-agnostic matching service. Refactored **out of**, not a modification to, `grandtour-reconciliation.mjs`'s `classifyRiderMatch`/`classifyTeamMatch` (those stay exactly as they were — they match an official stage *result's* rider rows against an already-tour-scoped roster; this is a new, separate service for matching a race *entry*, e.g. a startlist row, against the cross-race registry, before any tour-specific rider row necessarily even exists). Priority order, first confident/unambiguous tier wins: (1) an explicit UCI rider id already on the entry — a stale/unmatched id is itself a review item (`unmatched_uci_rider_id`), never silently falls through to name matching; (2) another trusted external id (pass-through only — no such id type exists in this codebase yet); (3) exact canonical `normalized_name` match; (4) exact alias match (only tried when the name tier didn't already resolve to exactly one candidate); (5) scored candidate — `scoreRaceEntryCandidate` generalizes `uci-match.mjs`'s tiers to operate on `uci_riders`' own field shape, and **a DOB conflict (both sides carry a DOB and disagree) always forces low confidence**, rejecting an automatic match regardless of how well name/nationality/team otherwise agree; more than one plausible (high/medium) scored candidate degrades to review (`ambiguous_candidate`), never picks the first; (6) manual review queue. Race-specific fields (bib, entered team/nationality) are captured in the result's `sourceEntry` for record-keeping but are **never** written back onto the canonical `uci_riders` row.
- **`uci-rider-sync-supabase.mjs`** — thin DB layer mirroring `grandtour-reconciliation-supabase.mjs` (anon-key reads)/`grandtour-apply.mjs` (service-role writes)'s existing split. Read functions accept whichever client the caller supplies; write functions (`applyRegistryPlan`, `applyAliasInserts`, `applyTeamHistoryPlan`, `applySpecialtyPlan`, `insertReviewItems`, `insertSyncRun`/`updateSyncRun`) are only ever called from the CLI's `--apply` path.
- **`uci-rider-sync.mjs`** — the main sync CLI. `--year`/`--discipline`/`--page`/`--from-page`/`--to-page`/`--search`/`--rider-id`/`--dry-run` (default)/`--apply`/`--write-files`/`--refresh-cache`/`--compare-production`/`--confirm-production`, same hand-rolled arg-parser + `decodeJwtRole`/`isProductionSupabaseUrl` safety-gate pattern as `grandtour-apply.mjs`. `fetchAllUciListingPages` paginates `GET /api/riders/<discipline>/<year>?page=<n>` following the response's **own** `totalItems`/`pageSize` — never a hardcoded page count — and dedupes by `uciRiderId` across pages. `shouldFetchUciProfile` skips a profile fetch (saving a network round trip) whenever every listing-visible field (given/family name, nationality, current team) already matches the stored row **and** a DOB is already known; a brand-new rider or one still missing a DOB always gets fetched. Because `uci_rider_id` is the entry's own confirmed identity (not a fuzzy candidate being scored against an independent source, unlike the Tour importer's UCI *search* match), a successfully-fetched profile's DOB/nationality is trusted at `matchConfidence: "high"` for the registry merge gate. Staged: every plan (registry + alias + team-history + specialty + review items) is built in full before any write happens, so a mid-run failure can't leave a half-applied, misleadingly-partial state — `uci_rider_sync_runs.status` (`completed`/`failed`/`partial`) records exactly what happened. `--compare-production` is a **designed-but-unexercised** flag: it would read a second, explicitly-separate production Supabase (via its own `UCI_COMPARE_PRODUCTION_URL`/`UCI_COMPARE_PRODUCTION_ANON_KEY` env vars, anon-key read-only, gated by the same `isProductionSupabaseUrl`/`--confirm-production` pattern) purely to report a diff — never write anything anywhere — but was never actually run against a real production URL this session, per this task's own local-only constraint.
- **`uci-rider-review.mjs`** — small CLI: `--list [--status <status>]` (default `pending`), or `--resolve <id> --resolve-status <status> [--note <text>] [--create-alias <text> --alias-type <type> --alias-rider-id <id>] [--resolved-by <user-id>]`, calling `resolve_uci_rider_review_item` via a service-role client (same safety gate as the sync CLI).

## GrandTour official-letour feed pipeline

### Purpose
Import official Tour de France stage results and jersey holders from letour.fr into Supabase, safely, with a fully-gated dry-run → reconciliation → apply → admin-review → finalize → score flow. Everything defaults to read-only; writes require explicit multi-flag confirmation, and every write step (apply, admin-check, finalize) is a separate, independently-gated action. Apply, admin-check, and finalize are all callable by service_role (CLI) **or** by an authenticated cycling admin's own session (admin UI, since `20260714010000_grandtour_apply_authenticated_grant.sql` extended apply the same way `20260710060000` already extended admin-check/finalize) — score has always required a real authenticated admin session, never service_role. No route or RPC in this pipeline ever accepts/uses a service-role key from the browser or from Vercel.

### Pipeline stages
1. **Parse** (`scripts/grandtour-feed-provider.mjs`) — fetches `https://www.letour.fr/en/rankings/stage-<N>` and regex-parses:
   - The stage-result ranking table (`rankingTable` / `rankingTables__row` markup) → top finishers with position, name, bib, team, time, gap. This markup is confirmed against a real letour.fr page.
   - Jersey holders: **not** on the initial page. The four cumulative classifications that determine jersey holders (yellow=individual/GC, green=points, kom=climber, white=youth) live behind a "General ranking" tab whose four sub-tab URLs are embedded as a `data-ajax-stack` JSON attribute on the page (`extractGeneralClassificationAjaxUrls`). `fetchLetourJerseyHolders` scrapes those URLs fresh per stage, fetches each fragment (same `rankingTable` row markup, confirmed real), and takes the position-1 leader of each (`parseLetourClassificationLeader`). This is confirmed against real letour.fr markup (verified mid-session; the URL hashes are per-page-load tokens and must always be scraped fresh, never hardcoded).
   - `OfficialLetourGrandTourFeedProvider` orchestrates one main-page fetch + up to 4 jersey-classification fetches per stage, and attaches `jersey_holders` + `jersey_fetch_metadata` diagnostics (status: `found`/`table_not_found`/`empty_table`/`fetch_error`/`unsupported_markup`) to each parsed `stageResult`.
2. **Reconcile** (`scripts/grandtour-reconciliation.mjs` + `grandtour-reconciliation-supabase.mjs`) — read-only (anon key), matches parsed riders/teams against `grandtour_riders`/`grandtour_teams`/`grandtour_stage_startlists` by bib number first, then normalized name. Produces `reconcileStageResult()` output including `matchedRiders`, `jerseyHolders`, `safeToApply`, and `blockers`.
3. **Apply** — two paths, same RPC. CLI: `scripts/grandtour-apply.mjs` + `scripts/grandtour-feed-import.mjs`'s `runApply` reads a previously-generated `--reconcile` report from disk (never re-fetches live), validates every gate, then calls `apply_grandtour_official_stage_result` via `SUPABASE_SERVICE_ROLE_KEY`. Admin UI: the stage card's "Apply Official Result" button (only enabled after a passing "Run Official Check") calls `apps/mobile/api/admin/grandtour/apply-official-result.mjs`, which fetches a **fresh** report server-side (never trusts a client-supplied one), re-validates it with the exact same `scripts/grandtour-apply.mjs` functions the CLI uses, then calls the same RPC using the admin's own session (never a service-role key). Both paths write a **draft** (`is_final: false`, `review_status: 'imported'`, `source_mode: 'official_feed'`) `grandtour_stage_results` + `grandtour_stage_result_lines` (top-10 only) + `grandtour_stage_jersey_holders` (all 4, if provided). Never scores, never finalizes.
4. **Admin check** (`public.mark_grandtour_stage_result_checked` RPC — CLI: `scripts/grandtour-admin-stage.mjs --mark-checked`; UI: `/admin/grandtour-stages` "Mark Checked"; or direct SQL/service-role) — a human confirms the applied draft is correct. Sets `review_status: 'admin_checked'`. Required before finalize will accept the stage.
5. **Finalize** (`public.finalize_grandtour_stage_result` RPC) — sets `is_final: true`, `review_status: 'finalised'`. Refuses unless `review_status = 'admin_checked'` first. Never scores, never touches result lines/jersey holders.
6. **Score** (`public.recalculate_grandtour_stage_scores` — pre-existing, unmodified) — requires an **authenticated cycling-admin session** (not service_role — it's `security invoker` and checks `auth.uid()` against `user_app_memberships`). Only works once `is_final = true`.

Distinct from the pre-existing generic `grandtour_game_audit` trigger-based trail (still fires independently on every raw row mutation for a different purpose — both exist, on purpose).
- **Trigger gotcha (fixed, keep in mind for future append-only tables):** `grandtour_result_audit_log`'s FK columns (`stage_result_id`, `changed_by`) use `on delete set null`. An append-only trigger that unconditionally blocks all `UPDATE`s will also block that legitimate FK-driven `SET NULL` action when the referenced row is deleted — breaking any documented delete/rollback path for the parent table. The fix: explicitly allow updates that *only* null out an FK column and change nothing else; reject everything else.

### RPCs
- `public.apply_grandtour_official_stage_result(p_stage_id, p_result_lines, p_reconciliation, p_dry_run_status, p_source, p_finalize, p_reason, p_request_id, p_jersey_holders, p_team_result_lines)` returns jsonb — `security definer`. Re-validates the reconciliation payload server-side, refuses `p_finalize: true` unconditionally, upserts jersey holders via `on conflict (stage_id, jersey_type) do update`, idempotent on unchanged result-line content (`no_change` vs `applied`). **TTT stages are refused unless `grandtour_stages.ttt_timing_rule = 'individual_time'`** (see "GrandTour TTT (individual_time) team-result support" above) — in that one supported case, `p_result_lines` must be empty and `p_team_result_lines` (added `20260714020000_grandtour_apply_ttt_individual_time_result.sql`) supplies the team result instead. Tags new rows `review_status='imported'`, `source_mode='official_feed'`; logs one `official_import_applied` row to `grandtour_result_audit_log` on a genuine `applied` outcome (not on `no_change`; `changed_by` on that row is always `null` regardless of caller — there's no `p_applied_by` parameter, unlike mark-checked/finalize's explicit `p_checked_by`/`p_finalized_by` — a known, intentionally-left-alone gap). **Callable by `service_role` (CLI) or by an authenticated cycling admin's own session (admin UI) since `20260714010000_grandtour_apply_authenticated_grant.sql`** (same `auth.role() = 'service_role' or is_cycling_admin()` guard pattern as mark-checked/finalize).
- `public.mark_grandtour_stage_result_checked(p_stage_id, p_checked_by, p_note, p_request_id)` returns jsonb — `security definer`. Requires exactly 10 result lines + 4 jersey holders, non-TTT, a draft result must exist. Sets `review_status='admin_checked'`. Never scores, never finalizes. `p_checked_by` is FK'd to `auth.users(id)`. **Callable by `service_role` (no internal check needed) or by `authenticated` if `grandtour_private.is_cycling_admin()` is true** (internal guard: `auth.role() = 'service_role' or is_cycling_admin()`, added by `20260710060000_grandtour_admin_check_finalise_authenticated_grants.sql` alongside the `authenticated` EXECUTE grant, so the admin UI can call it directly without a service-role key).
- `public.finalize_grandtour_stage_result(p_stage_id, p_finalized_by, p_reason, p_request_id)` returns jsonb — `security definer`. **Signature changed mid-project** (originally 3-arg `(p_stage_id, p_reason, p_request_id)` with no review-status gate; redesigned to require `review_status='admin_checked'` first and take an explicit `p_finalized_by`). Sets `is_final=true`, `review_status='finalised'`. Idempotent (`no_change` if already final). Never scores. Same `service_role`-or-admin-`authenticated` grant/guard as `mark_grandtour_stage_result_checked` above, added by the same migration.
- `public.set_grandtour_manual_result_entry_enabled(p_grand_tour_id, p_enabled, p_changed_by, p_reason)` returns jsonb — `security definer`, `service_role`-only. Flips `grand_tours.manual_result_entry_enabled`; logs to `grandtour_game_audit` (`action='admin_override'`) since it's tour-scoped, not stage-scoped.
- `public.recalculate_grandtour_stage_scores(...)` — pre-existing, unmodified. `security invoker`, requires an authenticated cycling-admin session (checks `grandtour_private.is_cycling_admin()` via `auth.uid()`) — **the service-role key alone cannot call this**, unlike every other RPC above.
- `public.is_current_user_cycling_admin()` returns boolean — `security definer`, `set search_path = ''`. Added by `20260713010000_grandtour_is_current_user_cycling_admin_rpc.sql`. A public wrapper around the already-fixed `grandtour_private.is_cycling_admin()` (that private schema isn't exposed via PostgREST, so it couldn't be called directly). Granted to `authenticated` only — Postgres grants EXECUTE to `PUBLIC` by default on a new function, so this migration explicitly `revoke all ... from public` first, then grants to `authenticated` (the same gotcha bit the local SQL test on first pass: `anon` could otherwise call it despite no explicit grant). Read-only, no writes. Used server-side by `apps/mobile/api/admin/grandtour/run-official-check.mjs` to authorize before running any check, since that route has no service-role key available to it.
- `public.correct_grandtour_stage_result_from_reviewed_report(p_stage_id, p_result_lines, p_jersey_holders, p_reconciliation, p_reason, p_request_id)` returns jsonb — `security definer`. Added by `20260711010000_grandtour_correct_stage_result_rpc.sql` (Part C). Same `service_role`-or-admin-`authenticated` guard as mark-checked/finalise. **The only RPC allowed to touch an existing (draft or already-finalised/scored) result** — `apply_grandtour_official_stage_result` still refuses outright if a different draft or any final result already exists; this is the deliberate, audited bypass of that refusal. Refuses a blank `p_reason`, a stage with no existing result at all (use apply instead), TTT stages, and every apply-style reconciliation/line/jersey-holder gate. Idempotent no-op (`status='no_change'`, no writes at all) on byte-identical content. Never finalises, never scores. CLI: `scripts/grandtour-admin-stage.mjs --update-results --from-report <path> --reason <text>` (never fetches letour.fr itself — same `--from-report`-only trust boundary as `--apply`). ### Postgres gotchas learned this session (apply broadly)
- Adding a parameter to a `security definer` function via a plain `create or replace function` is **not** a same-OID replace if the parameter list changes — Postgres treats a different arg-type list as a distinct overload, silently leaving both old and new versions defined. Whenever a function's parameter list changes (not just its body), explicitly `drop function if exists ...(<old exact signature>)` before recreating it. `create or replace function` is only safe as a plain in-place update when the parameter list is byte-identical.
- **A `returns boolean` SQL function whose body is `select <expr> from ... where exists (...)` returns SQL `NULL` for the "false" case, not `false`** — if the `where exists(...)` filters out the row, the query returns *zero rows*, and a zero-row scalar-context function call evaluates to NULL. This silently broke `grandtour_private.is_cycling_admin()`: every `if not is_cycling_admin() then raise exception ...` guard (imperative PL/pgSQL) was a no-op for non-admins, because `if not NULL` is `NULL`, and `if` only takes the branch when the condition is `TRUE`. Confirmed via direct testing that a non-admin authenticated session could call `recalculate_grandtour_stage_scores()` and sail straight past the intended admin check. **RLS `using(...)`/`with check(...)` policies were unaffected** — Postgres RLS treats NULL the same as false — only the `if not X then raise` pattern was exposed. Fixed in `20260710060000_grandtour_admin_check_finalise_authenticated_grants.sql` by rewriting the function body as `select exists (...)`, which always returns a genuine `true`/`false`, never NULL. **Lesson: any `returns boolean` SQL function meant to be used in an imperative `if not fn() then raise` guard must be written as `select exists(...)` (or with an explicit `coalesce(..., false)`), never as a bare `where exists(...)`-filtered row select** — write a genuine negative-path test (a real non-admin session actually calling the guarded function) before trusting *any* such guard, not just a positive-path (admin-succeeds) test.
- **`grandtour_stage_result_lines`/`grandtour_stage_jersey_holders` both have "prevent final delete" triggers** (`grandtour_stage_result_lines_prevent_final_delete`/`grandtour_stage_jersey_holders_prevent_final_delete`, from `20260629080958_grandtour_mvp.sql`) that block `DELETE` while the parent `grandtour_stage_results.is_final` is true. `correct_grandtour_stage_result_from_reviewed_report` (Part C) therefore flips `is_final` to `false` **first**, in the same function call, before deleting/reinserting any line or jersey-holder row — reversing that order makes correcting an already-finalised stage fail outright with an opaque trigger error. Same applies to `grandtour_stage_team_result_lines` (TTT) via `prevent_final_team_result_line_delete`, not currently exercised since TTT corrections are out of scope.

### Safety conventions (load-bearing — don't relax without explicit instruction)
- **Dry-run by default everywhere.** `--apply` requires `--confirm-provider official-letour`, `--confirm-stage <N>`, and (against a known production URL) `--confirm-production`.
- Apply mode **never** re-fetches or re-reconciles live — it only trusts a `--from-report` file, checked for freshness (max 6 hours old) and full gate compliance (`validateReportForApply`).
- Apply is **one-stage-only**, **top-10-only**, and writes **drafts only**. Admin-check, finalize, and score are separate, later, independently-approved steps — never bundle them into the same operator sitting as apply without a fresh review.
- `SUPABASE_SERVICE_ROLE_KEY`'s JWT `role` claim is decoded and checked (`decodeJwtRole`) before any apply — refuses to run with an anon key even if misconfigured.
- Known production project ref: `nsdpilmmrfobiapbwona` (`KNOWN_PRODUCTION_PROJECT_REFS` in `scripts/grandtour-apply.mjs`).
- The scheduled GitHub Actions workflow (`.github/workflows/grandtour-daily-feed-dry-run.yml`) is dry-run only — no `SUPABASE_SERVICE_ROLE_KEY`, no `--apply`, ever.

### Team-name normalization
`normalizeTeamName` (in `scripts/tdf-data-utils.mjs`) has its own implementation (distinct from `normalizeRiderName`) that folds punctuation/hyphens/pipes to whitespace before comparing, so official-letour's compact uppercase names (`"LIDL-TREK"`) correctly equate to canonical DB names (`"Lidl - Trek"`). `normalizeRiderName` deliberately does not do this (hyphenated surnames must not be split).

### GrandTour TTT (`individual_time`) team-result support
**TTT is no longer unconditionally blocked.** Earlier documentation in this file (and the original code comments) said "TTT stages are never safe to apply" as an absolute rule — that was true for the *older* shared-block-time TTT rule, but real Tour de France 2026 Stage 1 turned out to use a different, real, documented UCI rule that this codebase already had unused schema support for (`grandtour_stages.ttt_timing_rule`, added `20260703025324_add_grandtour_ttt_schema_support.sql`, which had *already* tagged Stage 1 `individual_time` before this was ever investigated further).

- **The rule**: since the 2023 Paris–Nice, some TTTs are scored "N=1" — a team's official time is simply the time of the *first* rider from that team to cross the line; every other rider is timed individually from then on (source: Wikipedia's Team Time Trial article, corroborated against a live fetch of `letour.fr/en/rankings/stage-1` on 2026-07-14 — teammates' times differ by tens of seconds, not identical, exactly matching N=1 and ruling out the old shared-block-time rule).
- **letour.fr publishes no separate team-classification page for this** (confirmed: no `data-ajax-stack`/"Teams" tab exists alongside the four jersey sub-tabs) — the team result is instead *derived* from the same per-rider ranking table already parsed for every stage. `deriveTeamResultFromRiderRows` (`scripts/grandtour-reconciliation.mjs`) groups parsed riders by `team_name`, takes each team's minimum `parseLetourElapsedTime(row.time)` (new in `scripts/grandtour-feed-provider.mjs`, parses `"00h 21' 47''"` → seconds) as that team's official time, and ranks teams ascending. `reconcileTeamTimeTrialResult` then matches each derived team against `existingTeams` (reusing `classifyTeamMatch`) and reports blockers for anything unmatched/undetermined.
- **`reconcileStageResult` gates on `ttt_timing_rule`, not just `isTtt`.** New fields on its return value: `tttTimingRule` (read from `existingStage.tttTimingRule`, sourced by `fetchReconciliationContext` in `scripts/grandtour-reconciliation-supabase.mjs`, which now selects `ttt_timing_rule`) and `isSupportedTtt` (`true` only when `isTtt && tttTimingRule === "individual_time"`). Only an unsupported TTT (any other/null `ttt_timing_rule`) gets the hard blocker; a supported one is genuinely `safeToApply: true` once its derived team result and everything else reconciles cleanly. `tttTeamResult: { teams, blockers }` is always computed for any TTT stage (even unsupported ones, for visibility) but its blockers are only folded into the overall `blockers` array when supported.
- **Apply RPC**: `apply_grandtour_official_stage_result` gained a 10th parameter, `p_team_result_lines jsonb` (`{team_id, actual_position}[]`, same shape/rules as `p_result_lines` but for teams), in `20260714020000_grandtour_apply_ttt_individual_time_result.sql`. Branches on a freshly-read `grandtour_stages.ttt_timing_rule`: non-TTT and `team_time`/null-rule TTT stages behave exactly as before (rider lines only, TTT still unconditionally refused for the latter); `individual_time` TTT stages require `p_result_lines` empty and `p_team_result_lines` to have exactly 10 entries, each validated against `p_reconciliation.tttTeamResult.teams`, and write to `grandtour_stage_team_result_lines` (a table that already existed, unused, since `20260703025324`) instead. Same idempotency/no-change/different-draft-refused semantics as the rider path. **Signature change → explicit `drop function` + regranted `service_role`/`authenticated` EXECUTE** (a fresh `create function`, unlike a same-OID `create or replace`, resets to the default PUBLIC grant and drops prior grants entirely — see the Postgres gotcha below).
- **JS apply-mode layer**: `scripts/grandtour-apply.mjs` gained `selectTopNTeamResultLines(teams)` (same top-10-only v1 policy as `selectTopNRows`, but no separate matched-lookup pass needed since `tttTeamResult.teams` entries already carry a resolved `teamId`) and `validateReportForApply` now only enforces the old unconditional `isTtt`/`stageType` TTT refusal when `stage.isSupportedTtt !== true`. `buildApplyRpcParams` takes an optional `teamResultLines` and always includes `p_team_result_lines` (defaulting `[]`). Both `scripts/grandtour-feed-import.mjs`'s `runApply` (CLI) and `apps/mobile/api/admin/grandtour/apply-official-result.mjs` (admin UI server route) branch identically on `stage.isSupportedTtt` to build team lines vs. rider lines — kept in sync deliberately, per that route's own doc comment ("this route can never apply something the CLI path would refuse").
- **Scoring was already built** — `recalculate_grandtour_stage_scores` already branched on `stage_is_ttt` and scored `stage_top_5` picks against `grandtour_stage_team_result_lines` (via `20260703041335_implement_grandtour_ttt_scoring.sql`, which superseded an earlier draft of the same function in `20260703025324` — always check for the *latest* migration redefining a function, not just the first one you find), and the tip-selection schema already required `team_id` (not `rider_id`) for TTT stage picks. This was discovered, not built, this session — no scoring changes were needed. **The TTT scoring formula is a flat scale, not the rider path's tiered 10/8/6/4/2**: an exact-position team pick is 6 points, any other genuinely-top-5 team pick is a flat 3 points, and a team outside the top 5 (or with no result line) is 0 — plus a separate, independent "winning team bonus" of 4 points awarded whenever `predicted_position=1` correctly names the actual position-1 team (`team_result_available` must also be true — the top 5 team-result lines must exist — or every team pick scores 0/`pending`, distinct from the "not applied yet" case). Confirmed by an actual end-to-end local rehearsal (apply → mark-checked → finalize → a real submitted tip → score, `supabase/tests/grandtour_finalize_stage_result.sql` tests 20-24) with hand-computed expected points, not just "the RPC call succeeded."
- **Apply → admin-check → finalize was previously TTT-blocked at every layer, not just apply.** `mark_grandtour_stage_result_checked` and `finalize_grandtour_stage_result` both had their own separate, unconditional TTT refusals (`stage_type in ('team_time_trial','ttt')` → `raise exception`) — extending only the apply RPC (as the first pass in this session did) left the pipeline dead-ending after apply for every TTT stage, including `individual_time` ones, since scoring requires `is_final=true`, which was unreachable. Fixed the same way as apply, in `20260714030000_grandtour_ttt_individual_time_admin_review_workflow.sql`: both functions now read `ttt_timing_rule`, only refuse when it's not `individual_time`, and count `grandtour_stage_team_result_lines` instead of `grandtour_stage_result_lines` for the supported case. Both signatures were byte-identical to their existing versions, so this was a safe same-OID `create or replace` — no drop/regrant needed, unlike the apply RPC's own extension.
- **Postgres gotcha reconfirmed here** (same class as the `is_current_user_cycling_admin` one already documented below): dropping and recreating a `security definer` function (required whenever its parameter list changes) resets EXECUTE to the Postgres default (`PUBLIC`) and **loses every previously-granted role's access**, not just adds the new grant on top. Any signature-changing migration on an existing function needs its own explicit `revoke all ... from public, anon, authenticated` followed by `grant execute ... to <the actual intended roles>` — verified by rehearsing the SQL test suite locally, which is exactly how a first draft of this migration (missing the `service_role` regrant) would have shipped a function only `authenticated` could call.
- **Admin UI display (fixed)**: `packages/supabase-client/src/grandtourAdmin.ts`'s `listGrandTourStageAdminSummaries` now counts `grandtour_stage_team_result_lines` instead of `grandtour_stage_result_lines` for a TTT stage's `resultLineCount` (a local `isTttStageTypeValue` helper, not imported from the app layer — this package must not depend on `apps/mobile`). `getGrandTourStageAdminReviewDetails` takes a second `stageType` param and returns a new `teamLines: GrandTourAdminTeamResultLine[]` field (position + team name) instead of `lines` for a TTT stage. `GrandTourStageAdminCard.tsx` (`apps/mobile/components/`) renders whichever table applies, via a new `isTttStageType` pure helper in `apps/mobile/lib/grandtourAdminExperience.ts`. The "Latest official check" preview panel (`grandtourOfficialCheckExperience.ts`'s `summarizeOfficialCheckReport`) was fixed the same way — it used to build `topResultLines` from `parsedRiders` unconditionally, which for a TTT stage would show the top-10 *individual finishers* (misleading, not what actually gets applied); it now returns `topTeamLines` (from `tttTeamResult.teams`) and an `isTtt` flag instead, when the stage is TTT. `canMarkChecked`/`isStageDataComplete`/`getStageReviewWarnings` needed no changes — they already compared `resultLineCount` generically, agnostic to which table it came from. **Not touched, still rider-only-assumed**: the CLI (`scripts/grandtour-admin-stage.mjs`) and the correction RPC/workflow (`correct_grandtour_stage_result_from_reviewed_report`, the "Update Results" panel) — check both for the same gap before using them on a real TTT stage.
- **Verification**: 17 new/updated pgTAP-style scenarios in `supabase/tests/grandtour_apply_official_stage_result.sql` (unsupported-timing-rule TTT still refused, empty-rider-lines requirement, mismatched team/position rejection, happy-path apply, audit rows, idempotent reapply, changed-result rejection), plus 24 in `supabase/tests/grandtour_finalize_stage_result.sql` (the same coverage for mark-checked/finalize, ending in a full apply → admin-check → finalize → real submitted tip → score rehearsal with hand-checked point totals), all passing locally alongside the other 7 SQL test files (no regressions). ~15 new Node unit tests across `grandtour-feed-provider.test.mjs`/`grandtour-reconciliation.test.mjs`/`grandtour-reconciliation-supabase.test.mjs`/`grandtour-apply.test.mjs`. **Both migrations (`20260714020000`, `20260714030000`) have since been pushed to production** and their grants verified directly (`service_role`/`authenticated` can execute, `anon` cannot; exactly one 10-arg overload of `apply_grandtour_official_stage_result` exists, no leftover 9-arg duplicate) — see "Production state" below, now current as of this session.

### Duplicate stage-results emails (investigated, explained, not a bug)
Tony reported receiving 2 identical "Stage 12 results" emails. Investigated (code reading, no live DB access to that specific job row this session) and explained: this is **expected behavior of the event-driven notification dispatch** (see "Event-driven dispatch" above), not a bug. `grandtour_private.dispatch_stage_score_notifications` deliberately resets a stage's already-`sent` notification jobs back to `pending` on every `recalculate_grandtour_stage_scores` call (a genuine rescore/correction must re-notify people — this was itself a bug fix in an earlier session, see "Event-driven dispatch"), bumping `notification_generation` and minting a brand-new Resend `Idempotency-Key` (`stage-result:<stage>:<user>:g2`, etc.) each time. Both of the pipeline's dedupe layers (the `(user_id, stage_id, notification_type)` unique constraint, and Resend's own idempotency-key cache) are intentionally bypassed on a second scoring run, because that run is treated as a legitimately new email to send. If Stage 12 was scored/corrected twice in quick succession (an admin re-applying/re-running the correction workflow, a double-click in the admin UI, etc.), every participant correctly receives two structurally identical emails, each independently "Delivered." Job-claiming itself (`send-stage-results/index.ts`'s `claimBatch`) was checked and ruled out as a cause — it's a single atomic `UPDATE ... WHERE status='pending'`, so the 15-minute cron and the event-driven dispatch can never double-claim the same row even if they overlap. **Not fixed, since it isn't a bug**: if a future session wants to prevent this specific "user re-scores the same stage without changing the result" scenario from re-notifying, that's a new product decision (e.g. only bump `notification_generation` when the score actually changed) — not something this session changed unilaterally.

### Auth callback routing (fixed a stuck-spinner / stray-redirect bug on `/auth/callback`)
- **Problem this fixed:** opening the app or logging out could leave the browser sitting on `https://grandtour-three.vercel.app/auth/callback` with a permanent "Confirming your account…" spinner. Root cause: `AuthCallbackScreen` never looked at whether the URL actually carried a callback code/token/error — it just reflected `useAuth()`'s `loading`/`user` state and only redirected once `user` became truthy. Anyone who ever landed on that route without a live confirmation in progress (a stale tab, a bookmark, a browser-history revisit, or simply being on that route when they clicked "Log out") had no `user` and no URL error, so the screen span forever. Logout (`profile.tsx`) also never navigated anywhere after calling `signOut()`, relying entirely on the root `Stack.Protected` guard swap — which only redirects *away* from screens inside a protected group, and `/auth/callback` is deliberately outside both groups (reachable regardless of auth state), so it was never swept away by that mechanism.
- **Fix:**
  - `apps/mobile/lib/authCallbackExperience.ts` (new, pure, unit-tested — `apps/mobile/tests/authCallbackExperience.test.cjs`) — `decideAuthCallbackAction(params)` inspects the actual callback params (`code`, `error`/`error_description`, `access_token`+`refresh_token`) and returns exactly one of `show_error` / `exchange_code` / `set_session` / `redirect_home` — **`redirect_home` when none of those are present**, so "no valid callback params" always means an immediate redirect, never a spinner. `sanitizeInternalReturnPath(path)` only ever returns a same-origin relative path (rejects absolute URLs, `//protocol-relative`, `/\backslash-trick`, embedded `://` schemes, and any path that points back at `/auth/callback` itself, to prevent a redirect loop) — used for an optional `?next=` return path, always falling back to `/`.
  - `AuthCallbackScreen.tsx` rewritten around that decision: on `exchange_code` it calls `supabase.auth.exchangeCodeForSession(code)` itself (PKCE, used by the sign-up confirmation link); on `set_session` it parses `window.location.hash` (`parseHashParams`) and calls `supabase.auth.setSession({access_token, refresh_token})` (implicit-flow hash tokens, if ever used); on `show_error` it stops immediately and renders the error with a "Back to sign in" button; on `redirect_home` it redirects to `/` (or the sanitized `next`) with no spinner. A 15s timeout flips any stuck in-flight exchange to an error state — the screen can never spin forever, even on a hung network call. The effect runs exactly once per mount (`handledRef`), since a PKCE code and recovery hash tokens are single-use.
  - `packages/supabase-client/src/client.ts`: `detectSessionInUrl` changed from `typeof window !== "undefined"` to always `false`. This was required, not optional — leaving it `true` would race Supabase's own automatic URL-based session detection against `AuthCallbackScreen`'s explicit `exchangeCodeForSession` call over the *same* one-time-use code, and whichever lost the race would surface a false "Confirmation failed" even though the user was actually signed in. The web `/reset-password` recovery-link flow does **not** depend on `detectSessionInUrl` — `AuthProvider`'s `handleRecoveryUrl` already parses and applies those hash tokens independently via `Linking.getInitialURL()`/`setSession`, so it was unaffected. This app has no OAuth (`signInWithOAuth`) flow, which is the other thing `detectSessionInUrl` would normally matter for.
  - `apps/mobile/app/profile.tsx`: "Log out" now `await signOut()` then unconditionally `router.replace("/login")` in a `finally` (so it still redirects even if `signOut()` throws), instead of firing-and-forgetting `signOut()` with no navigation at all.
- **Not changed / deliberately out of scope:** `ResetPasswordScreen.tsx`'s own `signOut()` → `router.replace("/login")` after a password update was already explicit and correct; left untouched.

### Auth callback routing, part 2: fixed a production redirect loop after the above shipped
- **Problem this fixed:** even after the fix above, production still showed `/auth/callback` flashing "Confirming your account…" repeatedly instead of settling. The root cause was **two independent things deciding "where to navigate next" at the same moment**: (1) a successful `exchangeCodeForSession` call updates `AuthProvider`'s `session` state via `onAuthStateChange`, which flips the root `Stack.Protected` guards in `app/_layout.tsx` — and per Expo Router's own docs, a guard flip mutates navigation history and can redirect on its own; (2) `AuthCallbackScreen` was *also* calling `router.replace` at essentially the same instant, right after its own `exchangeCodeForSession(...).then(...)` resolved, without waiting for that guard state to actually settle first. Whichever "won" left the app in an inconsistent spot, and if the guard flip forced `AuthCallbackScreen` itself to remount while still on that route, its old per-mount `handledRef` reset to `false` and the effect fired again — issuing a **second** `exchangeCodeForSession` call against a code that was already being (or had just been) consumed by the first, which fails and re-enters the spinner/error cycle. Separately, `ProtectedRoute` was blanket-gating *every* route (including `/auth/callback`) behind `AuthProvider`'s own `loading` flag, briefly swapping in its own "Checking your session" spinner on top of `AuthCallbackScreen`'s — a second, unrelated spinner-flash layer.
- **Fix (single owner, deduped, and gated on the same signal the guards use):**
  - `apps/mobile/lib/authCallbackExperience.ts`: new `getAuthCallbackFlowKey(action)` — a stable string per distinct callback attempt (keyed on the actual code/token/error value). New `isAuthCallbackPathname(pathname)` — the one shared predicate for "is this the callback route", used by both `sanitizeInternalReturnPath` (unchanged behaviour, now implemented via this helper) and `ProtectedRoute` (new).
  - `AuthCallbackScreen.tsx` rewritten around a module-level (not component-state) `flowRegistry: Map<key, Promise<...>>`. Every mount for the *same* callback attempt — including a Strict Mode dev double-invoke or a guard-flip-forced remount — reuses the existing in-flight/completed promise instead of issuing a second Supabase call; only the currently-mounted instance acts on the result (`exchange it exactly once`, satisfied structurally, not by a ref that resets on remount). The screen also now has two phases: `working` (exchange in progress) → `awaiting_session` (exchange succeeded; waiting for `useAuth()`'s `user` to actually reflect it, bounded by a 5s fallback) → `redirecting` (calls `router.replace` exactly once, guarded by a ref). By construction, `AuthCallbackScreen` never navigates until the same session signal that drives the `Stack.Protected` guards has already been observed locally — so by the time it calls `router.replace`, the guards have already settled and there is nothing left to race.
  - `ProtectedRoute.tsx`: now reads `usePathname()` and skips its own loading-spinner gate entirely when `isAuthCallbackPathname(pathname)` is true — `/auth/callback` does its own independent session handling and must never be covered by the global "checking your session" spinner.
  - `apps/mobile/lib/authDebugLog.ts` (new): a trivial `__DEV__`-gated `console.log` helper. Added temporary tracing (pathname, callback **parameter names only, never values**, auth loading/session-exists booleans, `onAuthStateChange` event names, and every redirect's source+destination) in `AuthCallbackScreen`, `AuthProvider`, and `ProtectedRoute`, to make a future redirect-sequence issue diagnosable from a local dev console without shipping any console output to production (`__DEV__` is always false in a production/web-exported build).
  - **Verified, not changed:** the root route (`app/index.tsx` / `(auth)/index.tsx`) never redirects to `/auth/callback` under any auth state; nothing in the repo persists a `returnTo`/stale-pathname value anywhere (`localStorage`/`sessionStorage`/`AsyncStorage` — the only `AsyncStorage` usage is Supabase's own session storage); logout (`profile.tsx`) was already correct from part 1 (calls `signOut()` once, always navigates to `/login`, never `/auth/callback`).
- **Part 2's diagnosis turned out to be incomplete** — directionally correct about "two things racing to decide the next destination," but the actual, confirmed mechanism (part 3 below) was different and more fundamental. Deployed, still broken.

### Auth callback routing, part 3: the REAL root cause (confirmed with a real headless browser, not just code reading)
- **What changed:** parts 1 and 2 were both diagnosed from code reading alone (no browser tool was available). Part 2 shipped and the user reported it made **no difference** and had gotten *worse* — "any access to the main web page flashes and goes nowhere," not just `/auth/callback`. That breadth (every route, not just the callback one) was the tell that the real cause was structural, not specific to the callback screen. This time, Playwright (`chromium`) was installed locally and used to actually load the live production site and a local rebuild, with full console/`pageerror`/navigation-event capture - see the evidence below.
- **Bug A - a real, reproducible Expo Router error:** loading bare `https://grandtour-three.vercel.app/auth/callback` (no `code`, no error, no tokens) threw, in the browser console: `Error: Attempted to navigate before mounting the Root Layout component. Ensure the Root Layout component is rendering a Slot, or other navigator on the first render.` This is Expo Router's own documented error for calling an **imperative** navigation (`router.replace(...)`) inside a `useEffect` that can fire before the navigator has finished mounting - exactly what the `redirect_home` branch of `AuthCallbackScreen`'s first effect did, on literally its first possible tick.
- **Bug B - the actual production loop, confirmed with real event capture:** loading bare `https://grandtour-three.vercel.app/` produced **hundreds of same-URL client-side navigations per second, forever** (captured via Playwright's `framenavigated` listener: ~2 events every 5-6ms, no network requests after the first document load, so purely client-side). `https://grandtour-three.vercel.app/login` - a single, unambiguous route - loaded and settled cleanly in the same test. Root cause: **`app/index.tsx` and `app/(auth)/index.tsx` were both registered for the exact same URL path `/`** (Expo Router route groups like `(auth)` don't add a URL segment), differentiated only by which of two complementary `Stack.Protected` guards (`Boolean(user)...` vs `!user...`) was currently active. Two competing screens permanently claiming one path produced a permanent client-side navigation loop the moment an unauthenticated (or not-yet-resolved) session landed on `/`.
- **Fix:**
  - `app/(auth)/index.tsx` **removed** - it's now unreachable/unnecessary.
  - `app/_layout.tsx`: `index` moved out of the `Stack.Protected guard={Boolean(user)...}` group, registered unconditionally alongside `auth/callback` (same pattern: a screen that decides its own auth-gated content internally, rather than being swapped in/out by a guard). `/` now has exactly one owner.
  - `app/index.tsx`: the default export is now a thin guard - `if (!user || isPasswordRecovery) return <Redirect href="/login" />;` - before rendering the real dashboard (renamed to an internal `AuthenticatedDashboard` component). Uses the **declarative** `<Redirect>` component, not an imperative call.
  - `AuthCallbackScreen.tsx`: every code path that used to call `router.replace(...)` imperatively now instead sets `{status: "redirecting", destination}` and renders `<Redirect href={destination} />` in JSX - Expo Router's own documented-safe mechanism for "redirect once the navigator is ready," which structurally cannot throw the "before mounting Root Layout" error since it defers to the library's own render-time handling instead of firing a side effect at an arbitrary moment.
  - The "awaiting_session" gating from part 2 (wait for `useAuth()`'s `user` to reflect a successful exchange before redirecting) was kept - it's still correct and still prevents the callback screen from redirecting before AuthProvider's session state (and therefore the `Stack.Protected` guards) has caught up.
- **Verification method (record this pattern for future auth/routing bugs in this app):** `apps/mobile` has no browser automation tooling installed by default. `npx --yes playwright@latest install chromium --with-deps` downloads a working headless Chromium in this environment; a throwaway `npm init -y` + `npm install playwright@latest` in the scratchpad directory (never added to this repo's `package.json`) provides the `playwright` package for a driver script. A local static rebuild (`npm run web:build`) served via `serve -s dist` (SPA fallback, matching `vercel.json`'s catch-all rewrite) plus a Playwright script listening for `framenavigated`/`console`/`pageerror`/navigation-request events is what actually surfaced both bugs above - code reading alone had already missed the real cause twice.

### GrandTour auto dry-run admin notification email

### GrandTour auto dry-run: stage-resolution and exit-code fix
- **Problem this fixed:** a real scheduled run's `final-summary.json` showed `stageNumber: 1`, `finalStatus: "unsafe_review_required"` (blocked because "Stage is a TTT; official team-result source is not confirmed") — and the job failed (red X) because of it, and separately the mail step failed with `Error: Server address must be specified` (see above).
- **Root cause of the stage-1 selection (not a hardcoded fallback — a starvation bug):** `resolveAutomaticStage` (`scripts/grandtour-stage-calendar.mjs`) always picked the *earliest* eligible, not-yet-`isFinal` stage. A TTT stage can never become `isFinal` through this pipeline (the TTT safety rule in `scripts/grandtour-reconciliation.mjs` — preserved, unchanged, still covered by its own passing test `stage 1 TTT is never safe to apply even with a perfect match` — means it is never safe to apply, so it's never finalised). "Earliest eligible, not final" therefore re-selected the permanently-stuck TTT stage on *every single scheduled run forever*, even once later stages had raced and genuinely needed a first look. No literal `|| 1` fallback existed anywhere in the code — the selection logic itself produced `1` correctly, every day, by design, which was the actual bug.
- **Fix:** `resolveAutomaticStage` now prefers the **latest** eligible stage (`resolutionSource: "database_schedule"`); only if that latest one is already finalised (and `allowRerunCompleted` is false) does it fall back to the most recent still-unresolved older stage instead (`resolutionSource: "unresolved_stage"`) — so a genuine straggler is still self-healed, but a structurally-unresolvable early stage (like a TTT) can never again starve out attention from every later stage. Returns `resolutionSource: "none"` (never a hardcoded stage) when nothing is eligible. `resolveStageRange` (`scripts/grandtour-auto-dry-run.mjs`) adds a fourth source, `"manual_input"`, for explicit `--stage-number`/`--from-stage`/`--to-stage`, and now logs the resolution source, selected stage number, its start time, the current time, and the reason on every call.
- **Exit-code fix:** `computeExitCode` no longer treats `unsafe_review_required` as failure-by-default. A completed dry run with `finalError: null` and `finalStatus: "unsafe_review_required"` now **always** exits 0 (a `::warning title=GrandTour review required::<blockers>` annotation is printed instead) — it produced a real report; there's nothing to retry or fix in CI, only a human review to do. Exit 1 is reserved for genuine technical failures: any non-null `finalError`, plus `parser_drift`/`transient_failure_exhausted`/`configuration_error`/`unexpected_failure`. The `--fail-on-unsafe` CLI flag/workflow input is kept (parsed, still accepted) for backward compatibility but is now inert — it no longer affects the exit code.
- **Not changed by this fix:** the TTT safety rule itself (`scripts/grandtour-reconciliation.mjs`) — at the time of this fix, a TTT was still never safe to apply. **This was later revisited and partially relaxed in a subsequent session** — see "GrandTour TTT (individual_time) team-result support" above; a stage tagged `ttt_timing_rule='individual_time'` (real Stage 1) can now be safe to apply, but the resolution-starvation problem this section describes was about the stage-*selection* logic, not the safety gate, and remains fixed regardless.
- Tests: `scripts/grandtour-stage-calendar.test.mjs` and `scripts/grandtour-auto-dry-run.test.mjs` cover the latest-first/straggler-fallback/none resolution paths (including a dedicated "never defaults to stage 1" case), the unconditional `unsafe_review_required` → exit 0 behaviour, the GitHub warning annotation, and a genuine `finalError` still forcing exit 1; `scripts/grandtour-auto-dry-run-workflow.test.mjs` covers the new SMTP env-var mapping and both mail-step/skip-notice guards.

### GrandTour auto dry-run: stage-availability grace period lowered 12h → 8h
`DEFAULT_STAGE_AVAILABILITY_GRACE_HOURS` (`scripts/grandtour-stage-calendar.mjs`) and every hardcoded fallback of it in `.github/workflows/grandtour-auto-dry-run.yml` (the `workflow_dispatch` input default and both the scheduled-run and dispatch-fallback shell defaults — three separate spots, all kept in sync) changed from `12` to `8`. **Problem observed**: the scheduled run at 2026-07-14 18:16 UTC found `finalStatus: "no_eligible_stage"` and correctly sent no email, even though a real stage (production's Stage 10, `starts_at` 2026-07-14 12:00 UTC — confirmed via a direct anon-key REST query against production, since the local dev seed's stage dates are synthetic and unrelated to the real production schedule) had already run and almost certainly had results posted; it just hadn't cleared the old 12h grace window yet (only 6h17m had elapsed at run time). 8h leaves less of a lag for a scheduled run to notice a finished stage, at the cost of running a bit closer to when letour.fr might not have posted results yet (still self-heals on the next scheduled run either way, per the "latest eligible, never a hardcoded stage" design above). ### GrandTour auto dry-run: schedule realigned to stage-start + grace (8h → 7.5h, cron moved 17:17 → 19:30 UTC)
**Problem this fixed**: the workflow's fixed daily cron (`17:17 UTC`) and the `stage_availability_grace_hours` eligibility filter (`8`, from the section above) were never actually kept in sync as a pair — the cron time was chosen independently, long before the grace-hours value existed. Every real production stage starts at `12:00 UTC` (confirmed directly against every stage row in `grandtour_stages`), so `17:17 UTC` is only ~5-6h after a same-day stage's start — always short of the 8h grace window. This meant the scheduled run could *structurally never* pick up the current day's own stage on the day it raced; a run at 2026-07-15 18:19 UTC on stage 11's own race day returned `finalStatus: "no_eligible_stage"` for exactly this reason (confirmed live, not inferred). The workflow only ever "worked" by lagging a full day behind and picking the stage up on the *next* day's run once >24h had elapsed — the "self-heals on a later run" behaviour documented above, but relied upon daily as normal operation rather than as the rare-failure safety net it was designed to be.
- **Fix**: the cron and the grace-hours default must be read as one pair — `starts_at time-of-day + grace hours` — not tuned independently. Grace hours lowered `8` → `7.5`, and the cron moved `17 17 * * *` → `30 19 * * *` (19:30 UTC = 12:00 UTC + 7.5h), so a normal day's stage is eligible the instant the scheduled run checks it. Changed in lockstep, all three places documented in the section above (`DEFAULT_STAGE_AVAILABILITY_GRACE_HOURS`, the `workflow_dispatch` input default, both shell defaults) plus the one `cron:` line — six total edit points now, not three; `docs/GRANDTOUR_APPLY_MODE_PRODUCTION_READINESS_CHECKLIST.md` §17.1/§17.2 and `docs/deployment.md`'s scheduled-dry-run verification step updated to match.
- **The day-late "self-heals on a later run" catch-up remains in the code** (`resolveAutomaticStage`'s straggler fallback, [[grandtour-auto-dry-run]]) but is no longer expected to fire routinely — with the schedule correctly aligned, a normal day's stage resolves same-day, and the fallback goes back to being a genuine safety net for actual failures (letour.fr down, a transient outage) rather than papering over a permanently mistimed cron.
- Also fixed, while touching this: the key-files table above and `docs/GRANDTOUR_APPLY_MODE_PRODUCTION_READINESS_CHECKLIST.md` §17.2 both still said `resolveAutomaticStage` picks the **earliest** eligible stage — stale since the "stage-resolution and exit-code fix" section further above already changed that to **latest** (with the straggler fallback). Corrected both to match the actual code, unrelated to the schedule fix itself but caught in the same pass.

### `supabase/functions/send-test-email/index.ts`
A diagnostic function proving the chain works, not the production sender (see `send-stage-results` below for that). Behaviour:
- **Only `POST`** is accepted; every other method gets `405`.
- **Requires a genuine authenticated Supabase session**, not just the public anon/publishable key. Supabase's own platform-level `verify_jwt` (on by default, and left on here — never disabled for this function) only checks that *some* validly-signed JWT was presented, and the anon key itself is one, so it alone isn't a real auth boundary. The function adds its own check on top: it builds a Supabase client from `SUPABASE_URL`/`SUPABASE_ANON_KEY` (both auto-injected by the platform, not custom secrets) with the caller's `Authorization` header, calls `auth.getUser()`, and returns `401` unless that resolves to a real user — confirmed locally that an anon-key-only bearer token is correctly rejected at this layer, separate from and in addition to the platform's own JWT gate.
- Reads `RESEND_API_KEY`, `RESULTS_EMAIL_FROM`, `RESULTS_EMAIL_REPLY_TO` (optional — only included in the Resend payload when set), `APP_PUBLIC_URL` from environment variables only — never the request body, never hardcoded, never logged.
- Validates the request body has a plausible `to` email address (`400` if not, `400` on unparseable JSON).
- Calls the Resend HTTPS API directly via `fetch` (no SDK dependency added) with both an HTML and a plain-text body and a GrandTour-branded subject.
- Sanitises every failure path: a missing/invalid Resend key or unreachable provider never leaks the raw upstream error body, header contents, or the key itself — just a generic `502`/`500` message plus (for a non-2xx Resend response) the upstream HTTP status code.

### `send-stage-results`: production daily stage-results email (implemented)
A later session implemented the function the section above only designed. Full workflow: stage finalised + scored → `send-stage-results` (scheduled every 15 minutes via pg_cron) generates one job per eligible participant, idempotently, then claims/sends/records outcomes. Schema discovery was done first against real migrations/generated types, not assumed — see the exact table/column/enum names below; they don't all match the original design sketch above (e.g. the tips table is `grandtour_tips`, not `grandtour_stage_tips`).

**The reliable "final AND scored" condition** (no single flag exists for this): `grandtour_stage_results.is_final = true` (a DB CHECK constraint keeps this equal to `review_status = 'finalised'`) **AND** `exists(select 1 from grandtour_stage_scores where stage_id = ...)`. `finalize_grandtour_stage_result` refuses to run if scores already exist, so finalise always precedes score — never the reverse. There's no "cancelled" stage-result state in this schema; a stage that never gets finalised simply never becomes eligible, with no special case needed. `packages/.../send-stage-results/eligibility.ts`'s `isStageReadyForNotifications` encodes exactly this.

**New schema** (migrations `20260715030000` through `20260715090000`):
- `public.grandtour_notification_preferences` (`user_id` PK → `profiles.id`, `results_email_enabled boolean default true`, `timezone`) — one row per user, provisioned automatically by the existing signup trigger (`app_private.handle_new_auth_user()`, extended in `20260715030000` to also insert a default-enabled preference row) and backfilled for pre-existing profiles in the same migration. RLS: users read/update only their own row; no admin/service policy needed since `send-stage-results` reads it with the service-role key (bypasses RLS). Deliberately just the one results-email toggle — not a marketing/promotional consent store.
- `public.grandtour_stage_notification_jobs` (id, user_id, stage_id, notification_type `stage_results`, channel `email`, status `pending`/`processing`/`sent`/`failed`/`skipped`, attempt_count, scheduled_at, processing_started_at, sent_at, provider_message_id, last_error_code, next_attempt_at, idempotency_key, timestamps). Unique on `(user_id, stage_id, notification_type)` and on `idempotency_key` (`stage-result:<stage_id>:<user_id>`) — both belt-and-suspenders against a duplicate send. RLS enabled; only a "cycling admins can read" policy exists (`grandtour_private.is_cycling_admin()`) — no authenticated write policy at all, by design, since every write comes from the Edge Function's service-role client. Reuses `grandtour_stage_scores.score_details` as the immutable per-user snapshot the email renders from — no separate snapshot table, since that column is already written once by scoring and never recomputed.
- `public.retry_grandtour_stage_notification_job(p_job_id, p_retried_by)` — `security definer`, service_role-or-cycling-admin only. Only ever moves a job from `failed` back to `pending` (resets `attempt_count`/`next_attempt_at`/`last_error_code`); raises on any other status, so it can never touch a `sent` job or create a duplicate. A legitimate, narrowly-scoped use of `security definer` (same justified pattern as the existing admin-check/finalize RPCs), not a "casual" one — the underlying table has no other write path for a human to use at all.
- `20260715060000` enables `pg_cron`/`pg_net` (neither was used anywhere in this repo before) and schedules `process-grandtour-stage-result-emails` (`*/15 * * * *`) via `cron.schedule(...)` calling `net.http_post(...)` against a function URL + scheduler secret stored in **Supabase Vault** (`grandtour_notification_function_url`, `grandtour_notification_scheduler_secret`) — never the Resend key, which stays an Edge Function secret only. `cron.schedule()` upserts by job name, so re-running this migration never creates a duplicate job. Seeded with **local-safe placeholder values** (the real local Edge Functions URL, a placeholder secret) so `supabase db reset` never calls a hosted URL; production needs its own separate step to point these at the real deployed function URL and a real random secret (see "Production steps remaining" below) — not committed here, per the task's own environment-neutral-migration requirement.
- `20260715070000` and `20260715090000` are **real bugs found and fixed during local dry-running**, not planned work:
  - `20260715070000`: both new tables' `revoke all ... from public, anon;` also silently stripped `service_role`'s ambient access (every role implicitly inherits a `PUBLIC` grant unless one was never given) — the exact same class of bug as the pre-existing `public.profiles` grant gap documented above. Confirmed via `has_table_privilege('service_role', ..., 'select')` being false on the new tables vs. true on every pre-existing one, and a real local run failing with a genuine `42501 permission denied`, not an RLS-filtered empty result. Fixed with an explicit grant.
  - `20260715090000`: `get_grandtour_leaderboard_with_movement` was first extended (in `20260715080000`) to allow a `service_role` caller (needed because `send-stage-results` calls it server-side, with no user session, to get each recipient's rank/previous_rank) using `auth.role() = 'service_role' or ...` inline in a single boolean expression — but **Postgres does not guarantee left-to-right short-circuit evaluation of AND/OR operands** (documented Postgres behaviour), so the `grandtour_private.can_access_competition(...)` call could still execute for a service-role caller lacking `USAGE` on that schema, failing with `permission denied for schema grandtour_private`. Fixed by using PL/pgSQL's own `if/else` control flow instead, which *does* guarantee an unexecuted branch is skipped. Every existing authenticated-caller code path (the app's own leaderboard screen) is unaffected — identical checks still run for it.

### `supabase/functions/_shared/email/` — reusable email-rendering module
New shared code directory (none existed before). `render-stage-results.ts`'s `renderStageResultsEmail(data)` takes fully-resolved, already-authoritative data (never recomputes scoring/points itself) and returns `{ subject, preheader, html, text }`. Pure, no Deno/network APIs — unit-tested directly with `node --test` (imports use explicit `.ts` extensions so both Deno and Node's native TS stripping resolve them).
- **Subject**: `"Stage N results: You scored S points"`, plus `" and moved up/down P places"` only when there's a genuine, known directional movement — a `same` or unknown (`new`) previous rank never fabricates a "moved 0 places" clause (`movement.ts`'s `computeRankMovement`/`formatSubjectMovementClause`).
- **Movement badge**: `▲ N` / `▼ N` / `—` / `NEW` (`formatMovementBadge`) — an unknown previous rank is always `NEW`, never treated as zero movement.
- **Top 5 badges**: green "Exact" (correct rider, correct position), blue "Top 5" (right rider, wrong position), grey "Miss"/"–" (miss/not picked) — same semantics as the existing results-screen badges in `apps/mobile/lib/grandtourStageResultsExperience.ts`, reimplemented (not imported) since this runs in a different runtime/deploy target. Points always come from the tip's persisted `grandtour_stage_scores.score_details.top_five`, never recomputed.
- **CTA**: `"Tip Stage N+1"` deep-linking to `${APP_PUBLIC_URL}/stages/<next-stage-id>` when the next stage is still open (compares `locks_at`/`starts_at` against now); otherwise `"View Full Results"` → `${APP_PUBLIC_URL}/results`.
- **Escaping**: all user-generated text (display names, rider names) is HTML-escaped (`htmlEscape.ts`) before interpolation — verified with a script-tag/quote injection test.
- **Degrades gracefully**: missing leaderboard data, an unknown previous rank, and a missing next stage are all handled explicitly (omitted section / `NEW` / fallback CTA) rather than rendering broken markup.

### `supabase/functions/send-stage-results/index.ts` — the production processing function
Private/internal-only — never a browser-callable arbitrary-recipient endpoint (unlike `send-test-email`). Pure logic lives in sibling files (`eligibility.ts`, `retryPolicy.ts`, `auth.ts`), each independently unit-tested; `index.ts` itself does the Deno-specific I/O (Supabase service-role client, `fetch` to Resend) and is verified by actually running it locally, not by unit test.
- **Authentication**: a shared `DAILY_RESULTS_JOB_SECRET`, checked with a constant-time comparison (`auth.ts`'s `constantTimeEqual`) — never a normal user JWT. `supabase/config.toml` disables platform `verify_jwt` for *only* this function (`[functions.send-stage-results] verify_jwt = false`), exactly because this internal check replaces it, per this feature's own instruction never to use `--no-verify-jwt` without first implementing and validating a real internal secret. **Gotcha hit while testing locally**: passing `--no-verify-jwt` as a CLI flag to `functions serve` applies globally to *every* served function, not just the one named on the command line — it briefly disabled JWT verification on `send-test-email` too during local testing. Always rely on `config.toml`'s per-function `verify_jwt` entries when serving multiple functions locally; never pass the CLI flag.
- **Invocation**: `{ stage_id }` for one stage, or `{ mode: "process_ready_stages" }` for the scheduled/cron path (resolves every stage that's genuinely final+scored and not yet fully processed). An optional `{ dryRun: true }` renders sample emails and reports counts without calling Resend or mutating any job's status.
- **Process**: generate missing jobs idempotently (`on conflict (user_id, stage_id, notification_type) do nothing`) → recover jobs stuck in `processing` beyond 10 minutes (crash recovery, never touches `attempt_count` since a stuck claim isn't necessarily a failed send) → claim a bounded batch (25) of `pending` jobs ready to (re)try → render + send with bounded concurrency (5 at a time, never an unbounded `Promise.all`) → record `sent`/`failed`/rescheduled outcome.
- **Skip reasons** (`eligibility.ts`'s `classifyParticipant`): a job is still created (for audit visibility) but immediately marked `skipped` with `last_error_code` set to `notifications_disabled` or `no_email` — a user who never participated (no scored/corrected tip on the stage) never gets a job row at all, since job generation only ever iterates users who already have one.
- **Retry policy** (`retryPolicy.ts`): attempt 1 immediate, attempt 2 after 15 minutes, attempt 3 after 60 minutes, then `failed`. A Resend `429`/`5xx`/network failure is retryable; `400`/`401`/`403`/`404`/`422` are treated as permanent (invalid/suppressed recipient, bad request, permission) and fail immediately regardless of attempt count — never retried.
- **Response summary** never includes a recipient email address or per-user result detail — only aggregate counts (`stages_checked`, `jobs_created`, `sent`, `failed`, `skipped`, `retrying`).
- **Not handled yet, a known gap**: TTT (team time trial) stages. `buildEmailData` assumes rider-based Top 5 picks (`grandtour_tip_selections.rider_id`) uniformly; a TTT stage's team-based picks would need separate handling (mirroring the existing TTT branches elsewhere in this pipeline) before this function is safe to run against a real TTT stage's finalised result.

### Event-driven dispatch: scoring itself now triggers the email, not just the 15-minute cron
A later session found two real bugs from the cron-only design above, both confirmed with live production data, not just code reading: (1) rescoring a stage could never re-notify a user who already had a `sent`/`failed` job for it — the job-generation upsert is `ON CONFLICT (user_id, stage_id, notification_type) DO NOTHING`, so an already-notified stage stayed silently un-notified forever after a correction; (2) the cron's own invocation shape (`{mode: "process_ready_stages"}`) sweeps **every** finalised+scored stage with no scoping — so the very first time the pipeline ran against a real backlog of 9 already-completed stages, one event (stage 2 becoming ready) caused all 9 to be swept and emailed simultaneously, and `SEND_CONCURRENCY = 5` firing that whole backlog at once produced repeated Resend `429`s, permanently failing 3 jobs once their 3-attempt retry budget was exhausted.
- **Fix** (`20260716010000_grandtour_stage_score_notification_dispatch.sql`): `recalculate_grandtour_stage_scores` now calls a new `grandtour_private.dispatch_stage_score_notifications(p_stage_id)` at the end of scoring (only when `affected_count > 0`) — via `net.http_post` to the same Vault-stored function URL/secret the cron uses, but **always scoped to `{stage_id: p_stage_id}` alone**, never the sweep-everything shape. That function also resets exactly that stage's own `sent`/`failed` notification jobs back to `pending` (clearing `sent_at`/`provider_message_id`/`last_error_code`, resetting `attempt_count`), so a correction/rescore actually re-sends. A new `notification_generation` column (bumped on every reset) is folded into the Resend `Idempotency-Key` (`stage-result:<stage>:<user>:g<generation>`, generation 1 omits the suffix for backward compatibility) so a corrected email is never silently deduped against the original send by Resend's own idempotency cache. `eligibility.ts`'s `buildStageResultIdempotencyKey` gained an optional third `generation` parameter to match.
- **The 15-minute cron is kept, unchanged, as a fallback safety net only** — not the primary path any more. Its own sweep-everything behaviour is now expected to almost always find `jobs_created: 0` in normal operation, since the event-driven path already handles every real scoring event.
- **Same grant gotcha caught again** (already documented twice elsewhere in this file): the new function's default `PUBLIC` execute grant had to be explicitly revoked and re-granted to `authenticated`/`service_role` — `recalculate_grandtour_stage_scores` is `security invoker`, so it runs as the calling admin, not as this function's owner; `security definer` only elevates privileges *inside* the called function's own body, it grants the caller nothing. Caught locally (all 10 SQL test files failed with `permission denied for function dispatch_stage_score_notifications` on the first migration draft) before ever reaching production.
- **Verified locally**: a real dispatch call correctly reset only the targeted stage's terminal job (generation 2, new idempotency key, immediately re-claimable) while leaving an unrelated stage's job completely untouched; `net.http_post` was confirmed queued and attempted (a local `Couldn't connect to server` response, since no local Edge Function server was running — expected, not a bug). All 10 SQL test files, 66+7 root tests, and the Edge Function's own unit tests pass.

### Real production incident: `RESULTS_EMAIL_REPLY_TO` misconfigured, 2 of 3 stage-11 emails failed
The very first real use of the new event-driven dispatch (scoring Stage 11 for the first time) sent 3 emails: 1 succeeded, 1 hit a transient Resend `429` (correctly retried automatically), and 2 failed with `provider_422` and no further detail — `last_error_code` only ever stored the bare HTTP status, not Resend's own error message, making this undiagnosable from the DB alone. Fixed by adding `readProviderErrorMessage` to `supabase/functions/send-stage-results/index.ts`, which reads Resend's JSON error body (`message`/`name`, truncated to 300 chars) and folds it into `last_error_code` as `provider_422:<message>` — deployed immediately (`npx supabase functions deploy send-stage-results`), then the 2 failed jobs were retried via `retry_grandtour_stage_notification_job` to surface the real reason: **`Invalid \`reply_to\` field. The email address needs to follow the \`email@example.com\` or \`Name <email@example.com>\` format.`** The `RESULTS_EMAIL_REPLY_TO` secret had been set (per its `updated_at` timestamp) to a value that wasn't a valid email, sometime the same session. Only 2 of the 3 concurrent requests hit this — the third happened to be rejected by Resend's rate limiter first, before the `reply_to` field was ever validated, which is why the failures looked inconsistent across an identical batch. Fixed by setting `RESULTS_EMAIL_REPLY_TO` to `GrandTour Admin <admin@grangewest.com.au>` via `npx supabase secrets set` (an actual secret **value** was set at the user's explicit instruction in this one case — this remains the only kind of secret action this assistant takes: setting an exact value the user dictated, never generating, guessing, or reading one back); both jobs were then retried again and confirmed `status: sent` with real Resend message IDs.
- **Lesson for this pipeline specifically**: `last_error_code` should always carry the provider's own message for a permanent failure, not just the status code — the status alone was not enough to diagnose a real, reproducible production failure even with direct DB access.
- Committed separately (`e2fdc08`, pushed same session) since it was a standalone diagnostic improvement, not part of the dispatch migration itself.

## Working conventions in this repo
- Local Supabase: `npx supabase status` for connection details; local demo anon/service-role keys are well-known Supabase CLI defaults, not secrets.
- Always rehearse against **local Supabase only** before any production action; production actions require explicit user authorization in the task itself, and even then double-check any pasted key's `role` claim (JWT) before using it — don't assume a pasted "anon key" or "production key" is actually what it's labeled.
- `npm run test:data` runs all `scripts/*.test.mjs` via Node's built-in test runner. `supabase/tests/*.sql` are pgTAP-style scripts run via `docker exec ... psql -f`, each in its own `begin`/`rollback` transaction. `npm run grandtour:reconcile:local-smoke`, `npm run grandtour:apply:local-smoke`, and `npm run grandtour:admin-stage:local-smoke` are real-Supabase (local) end-to-end smoke tests, not mocked — the apply and admin-stage ones perform real writes and clean up after themselves (including any test `auth.users` rows they create via `signUp`). The admin-stage smoke test's `--grand-tour-name`/`--grand-tour-year` fixture values must match `supabase/seed.sql`'s local grand tour (`"GrandTour France 2026"`, 2026) — the CLI's own real-world default (`"Tour de France"`) is a production concept, not a local one. From `apps/mobile`: `npm run test:ui` (pure UI logic, compiled via `tsc` into `dist/mobile-tests` then run with `node --test`) and `npm run test:api` (the `apps/mobile/api/**/*.mjs` server routes, `node --test` directly — no `tsc` step, since `apps/mobile/tsconfig.json`'s `include` deliberately excludes `api/`).
- Root `npm test` now runs `packages/tipping-core`'s test **and** `packages/supabase-client`'s test (`node --test src/*.test.ts`) in sequence. `packages/supabase-client` has no build step — Node 24 runs `.test.ts` files directly via its native TypeScript stripping (unflagged since Node 23.6+), as long as intra-package imports use an explicit `.ts` extension (Node's ESM resolver requires it; `tsc`/bundler resolution doesn't need it and would reject it without `allowImportingTsExtensions`, hence root `tsconfig.json` now `exclude`s `**/*.test.ts` from the main typecheck rather than enabling that compiler option repo-wide).
- **Node-in-CI gotcha:** `@supabase/supabase-js`'s `createClient()` throws synchronously on Node 20 (`Node.js 20 detected without native WebSocket support`) because its Realtime client requires a WebSocket constructor and no `ws` package is installed as a fallback in this repo. Any GitHub Actions workflow that constructs a Supabase client (i.e. runs `--reconcile`, or anything else that calls `createClient`) needs `node-version: '22'` or higher — `grandtour-auto-dry-run.yml` hit this on its first real run (fixed by bumping `'20'` → `'22'` → `'24'`, alongside `actions/checkout@v5`/`actions/setup-node@v5`). `grandtour-daily-feed-dry-run.yml` never calls `--reconcile` so it was never exposed to this, but keep it in mind for any future workflow that does.
- **Postgres gotcha:** a newly-created function gets EXECUTE granted to `PUBLIC` by default. `grant execute on function ... to authenticated` alone does **not** revoke that — `anon` (or anyone) can still call it. Any function meant to be admin/authenticated-only needs an explicit `revoke all on function ... from public;` before the intended grant (see `20260713010000_grandtour_is_current_user_cycling_admin_rpc.sql`, and the SQL test that actually caught this by exercising the `anon` role, not just `authenticated`).
- **PL/pgSQL gotcha for test authors:** an `exception when others` block implicitly rolls back to a savepoint taken at the *start* of that block — so a deliberate `DELETE`/setup statement meant to persist past the exception must be a separate top-level statement, never inside the same `do $$ ... exception ... $$` block being tested.
- **`test:ui` gotcha found this session**: `apps/mobile/tests/stageClosureExperience.test.cjs` required `../../../dist/mobile-tests/stageClosureExperience.js`, but `stageClosureExperience.ts` was never actually listed in `package.json`'s `test:ui` `tsc` command — the test only "passed" off a stale, previously-compiled artifact sitting in `dist/mobile-tests/`, invisible unless that directory was deleted first. Always `rm -rf dist/mobile-tests` (or equivalent) before trusting a green `test:ui` run if you suspect a file might be missing from its file list. Separately: adding a `components/theme.ts` file to that same `tsc` invocation (to test `getRaceHeadingAccent`) changed the compiler's inferred common `rootDir` from `lib/` to the whole `apps/mobile` tree, silently nesting every other file's output under `dist/mobile-tests/lib/*.js` instead of the flat `dist/mobile-tests/*.js` every existing test's `require(...)` path assumes — breaking all of them at once. Fixed by keeping `getRaceHeadingAccent`'s real implementation in `lib/raceAccent.ts` (re-exported from `theme.ts` for callers) so the `test:ui` file list never needs anything outside `lib/`.
