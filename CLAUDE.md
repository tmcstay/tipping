## Project

`tipping-suite` ‚Äî a tipping/prediction app. This file currently documents only the **GrandTour (Tour de France) official-letour feed pipeline**, which is the area actively under development on branch `feature/grandtour-official-letour-provider`. Other parts of the app (auth, other sports/competitions) are not yet documented here.

## GrandTour official-letour feed pipeline

### Purpose
Import official Tour de France stage results and jersey holders from letour.fr into Supabase, safely, with a fully-gated dry-run ‚Üí reconciliation ‚Üí apply ‚Üí admin-review ‚Üí finalize ‚Üí score flow. Everything defaults to read-only; writes require explicit multi-flag confirmation, and every write step (apply, admin-check, finalize) is a separate, independently-gated action. Apply remains service_role-only; admin-check and finalize are callable by service_role (CLI) **or** by an authenticated cycling admin's own session (admin UI) ‚Äî score has always required a real authenticated admin session, never service_role.

### Pipeline stages
1. **Parse** (`scripts/grandtour-feed-provider.mjs`) ‚Äî fetches `https://www.letour.fr/en/rankings/stage-<N>` and regex-parses:
   - The stage-result ranking table (`rankingTable` / `rankingTables__row` markup) ‚Üí top finishers with position, name, bib, team, time, gap. This markup is confirmed against a real letour.fr page.
   - Jersey holders: **not** on the initial page. The four cumulative classifications that determine jersey holders (yellow=individual/GC, green=points, kom=climber, white=youth) live behind a "General ranking" tab whose four sub-tab URLs are embedded as a `data-ajax-stack` JSON attribute on the page (`extractGeneralClassificationAjaxUrls`). `fetchLetourJerseyHolders` scrapes those URLs fresh per stage, fetches each fragment (same `rankingTable` row markup, confirmed real), and takes the position-1 leader of each (`parseLetourClassificationLeader`). This is confirmed against real letour.fr markup (verified mid-session; the URL hashes are per-page-load tokens and must always be scraped fresh, never hardcoded).
   - `OfficialLetourGrandTourFeedProvider` orchestrates one main-page fetch + up to 4 jersey-classification fetches per stage, and attaches `jersey_holders` + `jersey_fetch_metadata` diagnostics (status: `found`/`table_not_found`/`empty_table`/`fetch_error`/`unsupported_markup`) to each parsed `stageResult`.
2. **Reconcile** (`scripts/grandtour-reconciliation.mjs` + `grandtour-reconciliation-supabase.mjs`) ‚Äî read-only (anon key), matches parsed riders/teams against `grandtour_riders`/`grandtour_teams`/`grandtour_stage_startlists` by bib number first, then normalized name. Produces `reconcileStageResult()` output including `matchedRiders`, `jerseyHolders`, `safeToApply`, and `blockers`.
3. **Apply** (`scripts/grandtour-apply.mjs` + `scripts/grandtour-feed-import.mjs`'s `runApply`) ‚Äî reads a previously-generated `--reconcile` report from disk (never re-fetches live), validates every gate, then calls `apply_grandtour_official_stage_result`. Writes a **draft** (`is_final: false`, `review_status: 'imported'`, `source_mode: 'official_feed'`) `grandtour_stage_results` + `grandtour_stage_result_lines` (top-10 only) + `grandtour_stage_jersey_holders` (all 4, if provided). Never scores, never finalizes.
4. **Admin check** (`public.mark_grandtour_stage_result_checked` RPC ‚Äî CLI: `scripts/grandtour-admin-stage.mjs --mark-checked`; UI: `/admin/grandtour-stages` "Mark Checked"; or direct SQL/service-role) ‚Äî a human confirms the applied draft is correct. Sets `review_status: 'admin_checked'`. Required before finalize will accept the stage.
5. **Finalize** (`public.finalize_grandtour_stage_result` RPC) ‚Äî sets `is_final: true`, `review_status: 'finalised'`. Refuses unless `review_status = 'admin_checked'` first. Never scores, never touches result lines/jersey holders.
6. **Score** (`public.recalculate_grandtour_stage_scores` ‚Äî pre-existing, unmodified) ‚Äî requires an **authenticated cycling-admin session** (not service_role ‚Äî it's `security invoker` and checks `auth.uid()` against `user_app_memberships`). Only works once `is_final = true`.

### Database (GrandTour-relevant tables)
- `grandtour_teams`, `grandtour_riders` (with `bib_number`, `is_active`, `status`/`status_changed_at`/`status_reason`, `specialities`), `grandtour_stage_startlists` (`status`: provisional/confirmed/withdrawn/reserve/dns/dnf/otl/suspended/excluded/unknown).
- `grandtour_stage_results` ‚Äî `is_final` boolean, plus (new): `review_status` enum (`draft`/`imported`/`review_required`/`admin_checked`/`finalised`/`correction_required`), `admin_checked_at`/`admin_checked_by`/`admin_check_note`, `finalised_at`/`finalised_by`/`finalisation_reason`, `source_mode` enum (`official_feed`/`manual_admin`/`mixed`/`test`). Check constraint: `is_final = (review_status = 'finalised')` ‚Äî both must move together.
- `grandtour_stage_result_lines` (top-10, unique per stage+position and stage+rider).
- `grandtour_stage_jersey_holders` (`stage_id`, `jersey_type` enum: yellow/green/kom/white, `rider_id`; unique on `(stage_id, jersey_type)`). Triggers `validate_jersey_holder()` (must be on stage startlist; stage result must exist and not be final) and `prevent_final_jersey_holder_delete()`.
- `grand_tours` ‚Äî added `manual_result_entry_enabled boolean default false` (admin-controlled gate; no manual-entry RPC exists yet to read it ‚Äî see Known gaps).
- `grandtour_feed_import_runs` / `grandtour_feed_snapshots` ‚Äî audit trail written by the apply RPC (pre-existing).
- **New**: `public.grandtour_result_audit_log` (id, stage_id, stage_result_id, action, changed_by, reason, before_payload, after_payload, created_at) ‚Äî append-only (blocked by trigger), RLS-read-gated to cycling admins. `action` check constraint: `official_import_applied`, `manual_result_created`, `manual_result_updated`, `jersey_holder_updated`, `admin_checked`, `finalised`, `unfinalised`, `scored` (only the first three action values are currently written by real code: `official_import_applied` by apply, `admin_checked`/`finalised` by their RPCs ‚Äî the rest are reserved for the not-yet-built manual-entry/unfinalize/scoring-audit paths). Distinct from the pre-existing generic `grandtour_game_audit` trigger-based trail (still fires independently on every raw row mutation for a different purpose ‚Äî both exist, on purpose).
- **Trigger gotcha (fixed, keep in mind for future append-only tables):** `grandtour_result_audit_log`'s FK columns (`stage_result_id`, `changed_by`) use `on delete set null`. An append-only trigger that unconditionally blocks all `UPDATE`s will also block that legitimate FK-driven `SET NULL` action when the referenced row is deleted ‚Äî breaking any documented delete/rollback path for the parent table. The fix: explicitly allow updates that *only* null out an FK column and change nothing else; reject everything else.

### RPCs
- `public.apply_grandtour_official_stage_result(p_stage_id, p_result_lines, p_reconciliation, p_dry_run_status, p_source, p_finalize, p_reason, p_request_id, p_jersey_holders)` returns jsonb ‚Äî `security definer`, `service_role`-only. Re-validates the reconciliation payload server-side, refuses TTT stages, refuses `p_finalize: true` unconditionally, upserts jersey holders via `on conflict (stage_id, jersey_type) do update`, idempotent on unchanged result-line content (`no_change` vs `applied`). Tags new rows `review_status='imported'`, `source_mode='official_feed'`; logs one `official_import_applied` row to `grandtour_result_audit_log` on a genuine `applied` outcome (not on `no_change`).
- `public.mark_grandtour_stage_result_checked(p_stage_id, p_checked_by, p_note, p_request_id)` returns jsonb ‚Äî `security definer`. Requires exactly 10 result lines + 4 jersey holders, non-TTT, a draft result must exist. Sets `review_status='admin_checked'`. Never scores, never finalizes. `p_checked_by` is FK'd to `auth.users(id)`. **Callable by `service_role` (no internal check needed) or by `authenticated` if `grandtour_private.is_cycling_admin()` is true** (internal guard: `auth.role() = 'service_role' or is_cycling_admin()`, added by `20260710060000_grandtour_admin_check_finalise_authenticated_grants.sql` alongside the `authenticated` EXECUTE grant, so the admin UI can call it directly without a service-role key).
- `public.finalize_grandtour_stage_result(p_stage_id, p_finalized_by, p_reason, p_request_id)` returns jsonb ‚Äî `security definer`. **Signature changed mid-project** (originally 3-arg `(p_stage_id, p_reason, p_request_id)` with no review-status gate; redesigned to require `review_status='admin_checked'` first and take an explicit `p_finalized_by`). Sets `is_final=true`, `review_status='finalised'`. Idempotent (`no_change` if already final). Never scores. Same `service_role`-or-admin-`authenticated` grant/guard as `mark_grandtour_stage_result_checked` above, added by the same migration.
- `public.set_grandtour_manual_result_entry_enabled(p_grand_tour_id, p_enabled, p_changed_by, p_reason)` returns jsonb ‚Äî `security definer`, `service_role`-only. Flips `grand_tours.manual_result_entry_enabled`; logs to `grandtour_game_audit` (`action='admin_override'`) since it's tour-scoped, not stage-scoped.
- `public.recalculate_grandtour_stage_scores(...)` ‚Äî pre-existing, unmodified. `security invoker`, requires an authenticated cycling-admin session (checks `grandtour_private.is_cycling_admin()` via `auth.uid()`) ‚Äî **the service-role key alone cannot call this**, unlike every other RPC above.

### Postgres gotchas learned this session (apply broadly)
- Adding a parameter to a `security definer` function via a plain `create or replace function` is **not** a same-OID replace if the parameter list changes ‚Äî Postgres treats a different arg-type list as a distinct overload, silently leaving both old and new versions defined. Whenever a function's parameter list changes (not just its body), explicitly `drop function if exists ...(<old exact signature>)` before recreating it. `create or replace function` is only safe as a plain in-place update when the parameter list is byte-identical.
- **A `returns boolean` SQL function whose body is `select <expr> from ... where exists (...)` returns SQL `NULL` for the "false" case, not `false`** ‚Äî if the `where exists(...)` filters out the row, the query returns *zero rows*, and a zero-row scalar-context function call evaluates to NULL. This silently broke `grandtour_private.is_cycling_admin()`: every `if not is_cycling_admin() then raise exception ...` guard (imperative PL/pgSQL) was a no-op for non-admins, because `if not NULL` is `NULL`, and `if` only takes the branch when the condition is `TRUE`. Confirmed via direct testing that a non-admin authenticated session could call `recalculate_grandtour_stage_scores()` and sail straight past the intended admin check. **RLS `using(...)`/`with check(...)` policies were unaffected** ‚Äî Postgres RLS treats NULL the same as false ‚Äî only the `if not X then raise` pattern was exposed. Fixed in `20260710060000_grandtour_admin_check_finalise_authenticated_grants.sql` by rewriting the function body as `select exists (...)`, which always returns a genuine `true`/`false`, never NULL. **Lesson: any `returns boolean` SQL function meant to be used in an imperative `if not fn() then raise` guard must be written as `select exists(...)` (or with an explicit `coalesce(..., false)`), never as a bare `where exists(...)`-filtered row select** ‚Äî write a genuine negative-path test (a real non-admin session actually calling the guarded function) before trusting *any* such guard, not just a positive-path (admin-succeeds) test.

### Safety conventions (load-bearing ‚Äî don't relax without explicit instruction)
- **Dry-run by default everywhere.** `--apply` requires `--confirm-provider official-letour`, `--confirm-stage <N>`, and (against a known production URL) `--confirm-production`.
- Apply mode **never** re-fetches or re-reconciles live ‚Äî it only trusts a `--from-report` file, checked for freshness (max 6 hours old) and full gate compliance (`validateReportForApply`).
- Apply is **one-stage-only**, **top-10-only**, and writes **drafts only**. Admin-check, finalize, and score are separate, later, independently-approved steps ‚Äî never bundle them into the same operator sitting as apply without a fresh review.
- `SUPABASE_SERVICE_ROLE_KEY`'s JWT `role` claim is decoded and checked (`decodeJwtRole`) before any apply ‚Äî refuses to run with an anon key even if misconfigured.
- Known production project ref: `nsdpilmmrfobiapbwona` (`KNOWN_PRODUCTION_PROJECT_REFS` in `scripts/grandtour-apply.mjs`).
- The scheduled GitHub Actions workflow (`.github/workflows/grandtour-daily-feed-dry-run.yml`) is dry-run only ‚Äî no `SUPABASE_SERVICE_ROLE_KEY`, no `--apply`, ever.

### Team-name normalization
`normalizeTeamName` (in `scripts/tdf-data-utils.mjs`) has its own implementation (distinct from `normalizeRiderName`) that folds punctuation/hyphens/pipes to whitespace before comparing, so official-letour's compact uppercase names (`"LIDL-TREK"`) correctly equate to canonical DB names (`"Lidl - Trek"`). `normalizeRiderName` deliberately does not do this (hyphenated surnames must not be split).

### Known gaps / follow-ups
- **Manual result entry is designed but not implemented.** `grand_tours.manual_result_entry_enabled` and `set_grandtour_manual_result_entry_enabled` exist and work, but nothing reads the flag ‚Äî there's no RPC yet for an admin to hand-enter/correct a stage result when the feed fails. Full intended contract is documented in `docs/GRANDTOUR_APPLY_MODE_PRODUCTION_READINESS_CHECKLIST.md` ¬ß14.5.
- **No "unfinalise" RPC.** `'unfinalised'` is reserved in `grandtour_result_audit_log`'s action list but nothing writes it. A final result cannot currently be reopened via RPC (only by mistake-prone direct table access, which auto-marks a game-audit trail via the pre-existing generic trigger but not the new stage-workflow one).
- **`recalculate_grandtour_stage_scores` does not write to the new `grandtour_result_audit_log`.** It keeps using its own pre-existing `grandtour_game_audit` trail (`action='score_calculated'`/`'score_recalculated'`). `'scored'` is reserved in the new table's action list but currently unwritten ‚Äî left alone deliberately (it's a well-tested pre-existing RPC, out of this session's explicit scope).
- No production-readiness checklist exists for `scripts/load-tdf-2026-startlist.mjs` specifically (unlike the apply-RPC one). It also has no `decodeJwtRole`-style key-type guard.
- No correction workflow exists for an already-applied draft result with *different* content ‚Äî the apply RPC hard-refuses and requires manual admin-RLS-path correction or delete+reapply (the documented rollback: `delete from grandtour_stage_results where id = ...`, which now correctly cascades/nulls the audit log rather than being blocked ‚Äî see the trigger gotcha above).

## Working conventions in this repo
- Local Supabase: `npx supabase status` for connection details; local demo anon/service-role keys are well-known Supabase CLI defaults, not secrets.
- Always rehearse against **local Supabase only** before any production action; production actions require explicit user authorization in the task itself, and even then double-check any pasted key's `role` claim (JWT) before using it ‚Äî don't assume a pasted "anon key" or "production key" is actually what it's labeled.
- `npm run test:data` runs all `scripts/*.test.mjs` via Node's built-in test runner. `supabase/tests/*.sql` are pgTAP-style scripts run via `docker exec ... psql -f`, each in its own `begin`/`rollback` transaction. `npm run grandtour:reconcile:local-smoke`, `npm run grandtour:apply:local-smoke`, and `npm run grandtour:admin-stage:local-smoke` are real-Supabase (local) end-to-end smoke tests, not mocked ‚Äî the apply and admin-stage ones perform real writes and clean up after themselves (including any test `auth.users` rows they create via `signUp`). The admin-stage smoke test's `--grand-tour-name`/`--grand-tour-year` fixture values must match `supabase/seed.sql`'s local grand tour (`"GrandTour France 2026"`, 2026) ‚Äî the CLI's own real-world default (`"Tour de France"`) is a production concept, not a local one.
- **PL/pgSQL gotcha for test authors:** an `exception when others` block implicitly rolls back to a savepoint taken at the *start* of that block ‚Äî so a deliberate `DELETE`/setup statement meant to persist past the exception must be a separate top-level statement, never inside the same `do $$ ... exception ... $$` block being tested.
- Docs for this pipeline live in `docs/grandtour-results-feed.md` (central operational doc), `docs/grandtour-apply-mode-spec.md` (design spec), `docs/GRANDTOUR_PRODUCTION_DEPLOYMENT_CHECKLIST.md` (schema/migration deploys), `docs/GRANDTOUR_APPLY_MODE_PRODUCTION_READINESS_CHECKLIST.md` (full apply ‚Üí admin-check ‚Üí finalize ‚Üí score production runbook, ¬ß14 covers the review workflow), `docs/GRANDTOUR_TDF_2026_STARTLIST_PRODUCTION_LOAD.md` (production load record).
